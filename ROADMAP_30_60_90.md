# План 30 / 60 / 90 дней

## Цель

За `90 дней` довести текущий стенд до состояния управляемой пилотной системы документооборота, в которой:

1. пользователи работают через понятный веб-интерфейс
2. руководители видят статус документов, задач и заявок
3. `Rukovoditel`, `middleware` и `NauDoc` работают согласованно
4. система готова к пилоту на реальных подразделениях

## Базовый принцип

1. `Rukovoditel` = ежедневная работа пользователей
2. `NauDoc` = документы, версии, согласование, архив
3. `middleware` = связка, статусы, интеграция

## Первые 30 дней

### Цель этапа

Собрать стабильный каркас системы и сделать ее понятной для пользователей.

### Результат к концу 30 дней

1. единая структура меню и сущностей уже работает
2. интерфейс выглядит как единый продукт
3. роли и статусы формализованы
4. базовые процессы заведены в систему
5. стенд можно показывать заказчику как рабочий макет

### Неделя 1

1. Утвердить итоговую карту сущностей:
   - проекты
   - поручения
   - заявки на обслуживание
   - карточки документов
   - база документов
   - заявки на МТЗ
2. Утвердить роли:
   - сотрудник
   - исполнитель
   - руководитель
   - канцелярия
   - администратор
3. Утвердить словарь статусов по каждой сущности.
4. Утвердить, что где является “главной правдой”:
   - документ и архив в `NauDoc`
   - операционная работа в `Rukovoditel`

### Неделя 2

1. Довести UI `Rukovoditel` до единого стиля.
2. Настроить меню по ролям.
3. Упростить формы для сотрудников.
4. Сформировать отдельные рабочие представления для руководителей.
5. Подготовить понятные названия, подсказки и статусы на русском языке.

### Неделя 3

1. Довести сущности:
   - проекты
   - задачи
   - заявки на обслуживание
   - карточки документов
   - МТЗ
2. Настроить обязательные поля.
3. Настроить связи между сущностями.
4. Настроить типовые фильтры и быстрый поиск.

### Неделя 4

1. Подготовить базовые dashboard-экраны:
   - мои задачи
   - мои документы
   - заявки в работе
   - документы на согласовании
   - просроченные поручения
2. Проверить сценарии создания записей вручную.
3. Подготовить короткую инструкцию по работе для тестовой группы.

### Контрольные точки 30 дней

1. Пользователь может создать заявку, задачу и карточку документа.
2. Руководитель видит список просрочек и активных согласований.
3. Интерфейс визуально не выглядит как “сырой legacy”.
4. Меню и формы не перегружены.

## Дни 31-60

### Цель этапа

Сделать реальную связку между рабочим контуром и документным контуром.

### Результат к концу 60 дней

1. `middleware` не просто хранит ссылки, а начинает управлять связями
2. карточка в `Rukovoditel` связана с документом в `NauDoc`
3. статусы между системами становятся понятными и управляемыми
4. появляется формальная логика движения документа

### Неделя 5

1. Описать таблицу соответствия статусов:
   - черновик
   - на согласовании
   - на утверждении
   - подписан
   - на ознакомлении
   - завершен
   - архив
2. Утвердить правила переходов по статусам.
3. Выделить типовые виды документов:
   - входящий
   - исходящий
   - внутренний
   - служебная записка
   - договор
   - приказ

### Неделя 6

1. Доработать `middleware`:
   - хранение расширенных связей
   - статус синхронизации
   - журнал операций
   - ошибки синхронизации
2. Добавить API-точки под создание/обновление связей.
3. Подготовить базу под автоматизацию.

### Неделя 7

1. Реализовать автосвязь:
   - заявка -> карточка
   - проект -> документ
   - задача -> документ при необходимости
2. Добавить единые ссылки на `NauDoc` из `Rukovoditel`.
3. Отобразить статус связанного документа в пользовательском интерфейсе.

### Неделя 8

1. Настроить типовые маршруты для документов.
2. Протестировать сценарии:
   - создание документа
   - согласование
   - утверждение
   - ознакомление
   - архивирование
3. Проверить, чтобы пользователь не делал двойной ввод вручную.

### Контрольные точки 60 дней

1. Из карточки в `Rukovoditel` можно перейти к документу в `NauDoc`.
2. По записи видно, есть ли документ и на каком он этапе.
3. Есть базовый журнал интеграции.
4. Операционный и документный контуры больше не живут отдельно.

## Дни 61-90

### Цель этапа

Довести систему до пилотной эксплуатации на реальных пользователях.

### Результат к концу 90 дней

1. готов пилотный контур
2. есть инструкции и регламенты
3. есть базовые бэкапы и серверная дисциплина
4. руководители получают полезные панели
5. можно запускать 1-2 подразделения

### Неделя 9

1. Собрать руководительские панели:
   - документы на согласовании
   - просроченные задачи
   - неисполненные заявки
   - проекты с риском по срокам
   - заявки на МТЗ по подразделениям
2. Подготовить фильтры для руководителей и канцелярии.

### Неделя 10

1. Добавить уведомления:
   - email по ключевым событиям
   - уведомления о просрочке
   - уведомления о согласовании
2. При необходимости добавить веб-форму подачи заявок.
3. Подготовить основу для `email -> заявка`.

### Неделя 11

1. Подготовить серверный контур:
   - reverse proxy
   - единый URL
   - `HTTPS`
   - backup `MariaDB`
   - backup `bridge.db`
   - backup данных `NauDoc`
2. Подготовить staging и check-list запуска.
3. Подготовить журнал ошибок и базовый мониторинг доступности.

### Неделя 12

1. Провести внутреннее тестирование.
2. Подготовить пилотную группу пользователей.
3. Подготовить:
   - инструкцию сотрудника
   - инструкцию руководителя
   - инструкцию администратора
4. Зафиксировать список замечаний на пост-пилот.

### Контрольные точки 90 дней

1. Система доступна по понятной ссылке и работает в браузере.
2. Базовые процессы документооборота реально проходят через систему.
3. Руководители видят статусы и просрочки.
4. Есть резервное копирование и минимальная эксплуатационная готовность.
5. Можно запускать пилот на реальных пользователях.

## Что делаем в параллель весь период

### Бизнес-часть

1. Уточняем реальные типы документов.
2. Уточняем реальные маршруты согласования.
3. Уточняем роли подразделений.
4. Чистим лишние поля и лишние шаги.

### UX

1. Упрощаем формы.
2. Делаем понятные названия.
3. Убираем лишние переходы между экранами.
4. Делаем интерфейс пригодным для не-технических пользователей.

### Техническая часть

1. Ведем журнал изменений.
2. Проверяем контейнеры после каждой доработки.
3. Следим, чтобы связка не ломала `legacy`-контур `NauDoc`.
4. Сразу оформляем эксплуатационные заметки.

---

## Что не делаем в эти 90 дней

1. Полный рефакторинг `NauDoc`
2. Полное переписывание на новый стек
3. Полный складской ERP вместо базового МТЗ
4. Сложный enterprise-чат
5. Большую кастомную микросервисную платформу

Это сознательное ограничение, чтобы не расползтись и реально получить рабочий результат.
