## 4. Связи между модулями

### 4.1. Основная бизнес-связка

Текущая логика платформы выстроена так:

1. Работа начинается в `Rukovoditel`
2. В процессе используются:
   - проекты
   - заявки
   - задачи
   - карточки документов
3. Карточка документа является центральной точкой работы с документом
4. Из карточки доступны:
   - переход в `ONLYOFFICE`
   - переход в `NauDoc`
5. `Bridge` хранит интеграционные связи и синхронизирует состояние

### 4.2. Связка по проектам

Проверенный сценарий:

- `Проект -> Карточка документа -> Bridge -> NauDoc`

Это означает:

- проект может иметь связанную карточку документа
- карточка документа хранит ссылку на официальный документ
- интеграционный слой знает об этой связи и может тянуть статусы обратно

### 4.3. Связка по заявкам

Проверенный сценарий:

- `Заявка -> Карточка документа -> Bridge -> NauDoc`

Это означает:

- заявка может быть связана с карточкой документа
- документный маршрут не живет отдельно от рабочего процесса
- статус и ссылка могут быть возвращены обратно в рабочий кабинет

### 4.5. Важная особенность маршрутов

Финальная проверка показала важный архитектурный момент:

- `Задачи` и `Обсуждения` у нас являются дочерними сущностями `Проектов`

Поэтому корректный пользовательский маршрут для них идет:

- через рабочие отчеты
- или в контексте конкретного проекта

После выравнивания маршрутов:

- админский и пользовательский аудит идут по логичной траектории
- убрано ложное ощущение поломки из-за неверного прямого URL

## 6. Что по ONLYOFFICE, DocSpace и Workspace

### Реально внедрено

Сейчас реально внедрен и проверен только:

- `ONLYOFFICE Docs`

Именно он используется как встроенный веб-редактор документа.

### Что пока не внедрено как живой внешний сервис

На текущем стенде не развернуты как отдельные реальные инстансы:

- `ONLYOFFICE DocSpace`
- `ONLYOFFICE Workspace`

То есть их нельзя честно считать уже работающей частью production-контура.

Правильная формулировка по состоянию на сегодня:

- `ONLYOFFICE Docs` — live и проверен
- `DocSpace` — целевой следующий этап
- `Workspace` — возможный дополнительный корпоративный контур, но не внедрен
