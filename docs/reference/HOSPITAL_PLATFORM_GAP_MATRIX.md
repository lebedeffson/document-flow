# Матрица доведения платформы до hospital production

Дата: `2026-03-27`

## 1. Короткий вывод

Текущая платформа уже работает как связанный контур:

- `Rukovoditel` — рабочее место сотрудников
- `NauDoc` — официальный документный контур
- `ONLYOFFICE Docs` — встроенный редактор
- `Bridge` — интеграция, контроль связей, sync-ошибки, GUI-маппинг статусов

Но это еще не hospital production.

Главная причина: сейчас это не единая hospital-платформа с единым каталогом пользователей и финальными медицинскими ролями, а рабочий интеграционный каркас, который нужно довести до production-модели учреждения.

---

## 2. Архитектура по компонентам

| Компонент | Что делает | Где хранит данные | Как связан с остальными | Готовность сейчас | Что нужно доделать |
|---|---|---|---|---|---|
| `Rukovoditel` | Основной UI, процессы, заявки, проекты, карточки документов, база документов, МТЗ | `MariaDB` | Через `Bridge` связан с `NauDoc`; из карточки открывает `ONLYOFFICE` | Высокая для demo/pilot | Довести hospital-роли, GUI-конструктор связей и полей, user-mode без техшума |
| `NauDoc` | Официальный контур документа, регистрация, архив, маршруты, справочники | `Data.fs` (`Zope`) | Получает и отдает документный статус/ссылки через `Bridge`; открывается как отдельный официальный контур | Средняя/высокая | Единый вход, hospital-маршруты, увязка пользователей, формализация статусов и ролей |
| `Bridge` | Хранит связи, sync-ошибки, retry, GUI-маппинг статусов | `SQLite bridge.db` | Работает между `Rukovoditel` и `NauDoc`; вызывает внутренние sync-job'ы | Высокая как интеграционный слой | GUI-конструктор связей сущностей, GUI-маппинг полей, ручная перевязка записей |
| `ONLYOFFICE Docs` | Совместное редактирование документа | Собственные сервисные данные + файл документа в `Rukovoditel` | Запускается из карточки документа; callback идет в `Rukovoditel` | Высокая | Определить production-модель версий, публикации финальной версии в `NauDoc`, мобильный сценарий |
| `gateway` | Единый внешний вход | Конфиг `nginx` | Публикует `/`, `/docs/`, `/bridge/`, `/office/` | Высокая | Production-домен, нормальный TLS, закрытие внутренних портов |
| `sync worker` | Регулярно запускает sync-скрипты | Нет отдельной БД | Запускает `sync_service_requests`, `sync_project_documents`, `pull_bridge_updates` | Высокая | Наблюдаемость, alarm-поведение, график выполнения для prod |

## 3. Это одна БД или нет

Нет.

Сейчас платформа работает как **одна система на уровне пользователя**, но **не как одна физическая БД**.

### Что где лежит

1. `Rukovoditel`
   - бизнес-данные рабочих модулей
   - пользователи рабочего кабинета
   - карточки документов
   - связи заявок, проектов и рабочих сущностей

2. `NauDoc`
   - официальный документ
   - маршруты, регистрация, архив
   - справочники legacy-контура

3. `Bridge`
   - интеграционные связи
   - статус синхронизации
   - журнал ошибок
   - правила GUI-маппинга статусов

4. `ONLYOFFICE`
   - редакторская сессия
   - callback сохранения
   - временная редакторская логика

### Практический вывод

Для пользователей это должен быть **один вход и один рабочий сценарий**, но технически внутри это несколько подсистем.

Это нормальная целевая схема для hospital production, если:

1. роли согласованы
2. ownership данных четко определен
3. sync надежен
4. единый вход и каталог пользователей реализованы

## 4. Кто владеет какими данными

Это нужно закрепить до production.

| Тип данных | Источник истины сейчас | Целевая production-модель |
|---|---|---|
| Рабочие заявки | `Rukovoditel` | `Rukovoditel` |
| Проекты и задачи | `Rukovoditel` | `Rukovoditel` |
| Рабочая карточка документа | `Rukovoditel` | `Rukovoditel` |
| Совместно редактируемый черновик | `ONLYOFFICE` через `Rukovoditel` | `ONLYOFFICE Docs` + карточка в `Rukovoditel` |
| Официальный зарегистрированный документ | `NauDoc` | `NauDoc` |
| Интеграционные связи и sync-state | `Bridge` | `Bridge` |
| Пользователи | Частично раздельно | Единый каталог / `LDAP` / `SSO` |

Целевая логика должна быть такой:

1. создаем рабочий процесс в `Rukovoditel`
2. редактируем документ через `ONLYOFFICE`
3. публикуем или регистрируем официальный документ в `NauDoc`
4. `Bridge` фиксирует связь, статусы и возвращает информацию обратно

## 5. Как сейчас коммуницируют модули

## 5.1 Заявка

`Rukovoditel: Заявка -> Карточка документа -> Bridge -> NauDoc`

Что уже работает:

1. заявка создается и живет в `Rukovoditel`
2. для нее создается или поддерживается карточка документа
3. связь записывается в `Bridge`
4. ссылка и статус из `NauDoc` могут быть возвращены обратно

## 5.2 Проект

`Rukovoditel: Проект -> Карточка документа -> Bridge -> NauDoc`

Что уже работает:

1. проект связан с карточкой документа
2. карточка знает `NauDoc`-ссылку
3. обратная синхронизация проходит

## 5.3 Документ

`Карточка документа -> ONLYOFFICE Docs -> callback -> Rukovoditel`

Что уже работает:

1. документ открывается из карточки
2. editor config строится корректно
3. callback сохранения работает
4. browser e2e проходит в `Firefox` и `Chromium`

## 5.4 Администратор

`Bridge -> журнал sync-ошибок -> retry -> sync-job`

Что уже работает:

1. ошибки пишутся в журнал
2. retry реально запускает sync-job
3. статус ошибки меняется на `resolved` или `retry_failed`
4. GUI-маппинг статусов уже управляется через `Bridge`

## 6. Роль преподавателя: что с ней делать

Сейчас в системе есть роль:

- `Руководитель / преподаватель`

Для больницы это переходное название и его нужно заменить.

### Целевое переименование

| Текущее название | Hospital-замена |
|---|---|
| `Руководитель / преподаватель` | `Заведующий отделением / руководитель подразделения` |
| `Исполнитель / сотрудник` | `Врач / сотрудник подразделения` |
| `Заявитель / пользователь` | `Регистратура / заявитель` или `Сотрудник-заявитель` |
| `Канцелярия / делопроизводитель` | `Канцелярия / делопроизводство` |

### Что это значит

Нужно менять не только подписи, но и сами сценарии:

1. кто согласует
2. кто утверждает
3. кто регистрирует
4. кто только видит статус
5. кто может редактировать черновик

## 7. Что уже готово для hospital production

## Готово

1. единая точка входа через gateway
2. рабочий `Rukovoditel`
3. живой `NauDoc`
4. живой `ONLYOFFICE Docs`
5. рабочий `Bridge`
6. базовые роли и рабочие модули
7. сквозные связи `заявка/проект -> документ -> NauDoc`
8. журнал sync-ошибок
9. retry sync
10. GUI-маппинг статусов
11. backup/restore-скрипты
12. полный verify-конвейер

## Не готово

1. единая учетная система пользователей
2. hospital-рольовая модель
3. hospital-маршруты документов
4. GUI-конструктор связей между сущностями
5. GUI-конструктор маппинга полей
6. ручная перевязка записей
7. production-домен, secrets, monitoring, staging
8. полноценный `SSO/LDAP`
9. развернутые `DocSpace` и `Workspace`

## 8. Что обязательно закрыть до hospital prod

## P0

1. Переименовать и довести роли под больницу.
2. Формализовать hospital-маршруты документов.
3. Определить ownership данных между `Rukovoditel`, `NauDoc`, `ONLYOFFICE`.
4. Довести пользователей до единого каталога или `LDAP/SSO`.
5. Довести GUI администратора:
   - связи между сущностями
   - маппинг полей
   - ручная перевязка
6. Подготовить production-контур:
   - домен
   - TLS
   - секреты
   - backup
   - restore drill
   - monitoring

## P1

1. `staging`
2. аудит чувствительных действий
3. массовая синхронизация пользователей и ролей
4. более глубокий no-code routing
5. внешние комнаты и вторичный контур collaboration

## 9. Что нужно делать следующим практически

Рекомендуемый порядок следующий.

### Шаг 1

Сделать hospital-роли и hospital-термины:

1. переименовать роли
2. привести меню и главную к роли больницы
3. убрать остатки университетской семантики

### Шаг 2

Сделать GUI-конструктор связей:

1. какая сущность с чем связана
2. какое поле используется
3. обязательна ли связь
4. какие действия показывать в карточке

### Шаг 3

Сделать GUI-маппинг полей:

1. поле заявки -> поле карточки
2. поле проекта -> поле карточки
3. поле карточки -> поле sync metadata
4. поле sync metadata -> обновление назад

### Шаг 4

Решить пользователей:

1. единый каталог
или
2. `LDAP/SSO`

### Шаг 5

Перевести стенд в hospital production-подобный контур:

1. внутренний адрес учреждения
2. production env
3. backup, restore, monitoring
4. пилот на одном подразделении

## 10. Итог

Сейчас у нас уже есть хороший рабочий каркас.

Это означает:

1. платформа связана
2. базовый рабочий процесс есть
3. редактор работает
4. `NauDoc` и `Rukovoditel` связаны через `Bridge`

Но до hospital production еще нужно довести:

1. hospital-роли
2. пользователей
3. маршруты
4. GUI-связи и маппинг полей
5. эксплуатационный production-контур

Эта матрица должна использоваться как опорная схема для:

1. команды разработки
2. работодателя
3. проектного плана внедрения
4. пилотного запуска в учреждении
