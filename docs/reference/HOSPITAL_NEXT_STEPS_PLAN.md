6. пилотный регламент эксплуатации

8. ограниченное live-внедрение `DocSpace` и `Workspace` для первой production-волны

## 2. Принцип дальнейшей разработки

Дальше нельзя идти “вширь” и одновременно пилить все подряд.

Нужно держать 3 приоритета:

1. сначала надежность и управляемость
2. потом бизнес-маршруты больницы
3. потом расширение экосистемы и удобство

То есть следующий этап — это не “еще красивее UI”, а:

1. убрать прод-риски
2. закрыть реальные hospital-сценарии
3. подготовить перенос на сервер учреждения

---

## 3. Wave 1 — Production Baseline

### Цель

Довести стенд до состояния, в котором его можно безопасно переносить на Linux-сервер больницы.

### Что входит

1. вынести production-переменные в `.env`
2. заменить demo-secrets на реальные
3. зафиксировать production-domain и public URLs
4. включить backup timer
5. сделать первый restore drill
6. закрыть лишние наружные порты
7. оформить базовый ops-runbook
8. подготовить production/staging URL для `DocSpace` и `Workspace`, если они входят в первую волну

### Что уже закрыто частично

1. `.env.example` уже добавлен
2. compose-файлы уже умеют брать env-переменные
3. есть `prod_readiness_audit.py`
4. есть `install_backup_timer.sh`
5. есть `generate_prod_env.sh`
6. первый локальный `restore_drill.sh` уже выполнен успешно
7. есть безопасная ротация `NauDoc` credentials с backup, rollback и обновлением `.env`
8. текущий `.env` проходит `prod_readiness_audit.py` без blocker'ов

### Что осталось доделать

1. создать и перенести реальный `.env` на сервер больницы
2. проверить запуск контейнеров на hospital `.env`
3. повторить backup + restore drill на staging-контуре больницы
4. прописать runbook для ИТ-службы
5. поднять ограниченные инстансы `DocSpace` и `Workspace` для первой волны, если они входят в operating model

### Критерий приемки

1. `prod_readiness_audit.py` не показывает blocker'ов на hospital `.env`
2. система поднимается из `.env`
3. backup проходит по таймеру
4. restore drill подтвержден на production-like контуре
5. если `DocSpace/Workspace` входят в первую волну, они доступны по согласованным URL и не ломают основной контур

## 4. Wave 2 — Identity and Access

### Цель

Довести пользователей и роли до hospital-модели.

### Что входит

1. заменить тестовую рольовую семантику на финальную hospital-модель
2. довести роли:
   - врач
   - заведующий отделением
   - старшая медсестра
   - регистратура
   - канцелярия
   - администрация
   - ИТ-администратор
3. связать каталог пользователей между `NauDoc` и `Rukovoditel`
4. подготовить `LDAP-first` или `SSO-first` внедрение

### Практический порядок

1. описать hospital role matrix
2. привязать меню и доступы к новым ролям
3. расширить каталог профилей в `Bridge`
4. добавить рекомендации/подсказки по сопоставлению пользователей
5. синхронизировать role/email/department metadata между контурами
6. затем выйти на `LDAP/SSO`

### Что уже закрыто частично

1. hospital role matrix уже заведена в модели и specs
2. `Bridge` уже показывает matched, needs_review и manual-match профили
3. exact-match по `username` уже работает
4. suggestion flow по отображаемому имени уже поддерживается
5. администратор уже может принять подсказку через GUI без правки БД
6. администратор уже может через GUI описывать источники идентификации и отделять hospital-роли от технических ролей источников
7. есть локальный `LDAP-first` runtime baseline с живым bind/test/sync
8. полный verify уже проверяет `identity integration`

### Критерий приемки

1. каждая роль входит и видит только свой контур
2. нет остаточной учебной/демо-семантики
3. у пользователей есть понятное сопоставление между контурами

## 5. Wave 3 — Document Routes

### Цель

Довести документный контур до реальных сценариев больницы.

### Что входит

1. закрепить типы hospital-документов
2. довести `Маршрут документа`
3. описать реальные сценарии:
   - входящая регистрация
   - внутренний приказ
   - служебная записка
   - медицинский документ
   - направление / маршрут пациента
   - закупка / договор
   - ознакомление персонала
4. привязать роли к шагам согласования

### Что нужно сделать в коде

1. закрепить маппинг статусов по маршрутам
2. довести правила sync между `Rukovoditel`, `Bridge` и `NauDoc`
3. определить, где заканчивается черновик и где появляется официальный документ

### Критерий приемки

1. для каждого маршрута есть понятный сценарий
2. у статусов нет двусмысленности
3. роль пользователя влияет на доступ и шаг процесса

## 6. Wave 4 — Deep Integration

### Цель

Сделать интеграцию не только обозримой, но и полноценно управляемой через GUI.

### Что входит

1. более умный manual relink
2. GUI-конструктор связей между сущностями
3. прозрачная диагностика того, что и куда синхронизируется
4. доведение базового `Bridge -> NauDoc` write-back до hospital business-логики

### Что уже есть как основа

1. GUI-маппинг статусов
2. GUI-маппинг полей
3. `naudoc_projection`
4. manual relink базового уровня
5. рабочий прямой `document_cards -> NauDoc` write-back
6. GUI-слой route definitions для hospital-маршрутов

### Что именно доделать

1. закрепить write-back под финальные hospital document routes
2. добавить ручной выбор сущности при relink
3. добавить подсказки, с чем лучше связать запись
4. логировать изменения интеграционных правил

### Критерий приемки

1. админ может управлять интеграцией без правки PHP
2. ручная коррекция связей понятна и безопасна
3. изменение полевого маппинга реально отражается в документном контуре

2. bind-порты сервисов вынесены в env
3. есть `monitoring_snapshot.py`

5. есть отдельный [STAGING_RUNBOOK.md](/home/lebedeffson/Code/Документооборот/docs/reference/STAGING_RUNBOOK.md)

1. есть staging, отделенный от production
2. релиз проходит через staging verify

## 8. Wave 6 — Pilot in Hospital

### Цель

Запустить контролируемый пилот на одном подразделении.

### Что входит

1. выбрать одно подразделение
2. загрузить реальных пользователей
3. включить 2–3 реальных document routes
4. провести обучение
5. собрать замечания
6. скорректировать права, тексты, сценарии

### Критерий приемки

1. сотрудники реально работают в системе
2. ошибки синхронизации обрабатываются штатно
3. минимум один маршрут проходит от черновика до официального документа

## 9. Порядок разработки без распыления

Рекомендуемый порядок:

1. `Wave 1` — production baseline
2. `Wave 2` — identity and access
3. `Wave 3` — document routes
4. `Wave 4` — deep integration
5. `Wave 5` — monitoring and staging
6. `Wave 6` — pilot

Если делать иначе, будет хаос:

1. либо красивый UI без production-надежности
2. либо интеграция без реальных hospital-сценариев
3. либо пилот без поддержки и диагностики
