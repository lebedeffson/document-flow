# Сценарии запуска: Первый День и Первая Неделя

Дата: `2026-03-30`

## Назначение

Этот документ фиксирует не только технический запуск, но и то, что пользователи реально должны уметь сделать:

1. в первый день запуска
2. в первую неделю запуска
3. в конфликтных и пограничных случаях

## Честный вывод по железу

При полном same-host профиле `DocFlow + ONLYOFFICE Docs + DocSpace + Workspace` текущее тестовое железо показало высокий memory pressure:

1. `14.8 GiB RAM`
2. `7.4 GiB swap`
3. swap заполняется полностью
4. `Workspace Elasticsearch` на таком профиле нестабилен

Практический вывод:

1. `16 GiB` подходит для осторожного первого запуска
2. для безопасного hospital-prod лучше `32 GiB`
3. при `16 GiB` первую неделю надо запускать бережно, без агрессивного включения лишних модулей

## Рекомендуемый профиль для `16 GiB`

### Первый день

Обязательный рабочий контур:

1. `Rukovoditel`
2. `Bridge`
3. `NauDoc`
4. `ONLYOFFICE Docs`

Office-добавки:

1. `DocSpace` допустим как дополнительный контур, но не как обязательный пользовательский маршрут
2. `Workspace` не должен быть критичным для основной работы
3. основной пользовательский путь должен оставаться в `Rukovoditel + ONLYOFFICE Docs`

### Первая неделя

Нормальный режим:

1. пользователи работают в основном контуре
2. идет наблюдение за памятью, swap и health
3. `DocSpace/Workspace` не должны становиться blocker'ом для рабочих документов

## Сценарии первого дня

### 1. Администратор платформы

Должен уметь:

1. войти в систему
2. увидеть главную и контрольные разделы
3. открыть пользователей
4. создать нового пользователя
5. открыть `Bridge`
6. проверить состояние интеграции

### 2. Врач / сотрудник

Должен уметь:

1. войти под своей ролью
2. открыть карточку документа
3. открыть существующий Word-документ в `ONLYOFFICE`
4. изменить документ
5. сохранить изменения
6. снова открыть документ и увидеть, что изменения сохранились

### 3. Заведующий отделением

Должен уметь:

1. войти под своей ролью
2. открыть Excel-таблицу дежурств
3. изменить данные
4. сохранить изменения
5. открыть отчетный или контрольный маршрут
6. увидеть документ на согласовании

### 4. Старшая медсестра / координатор

Должна уметь:

1. войти под своей ролью
2. открыть документы отделения
3. открыть таблицу или карточку документа
4. работать с маршрутами без системных прав

### 5. Регистратура

Должна уметь:

1. войти под своей ролью
2. создать заявку
3. видеть только свои и связанные записи
4. открыть рабочую карточку документа по связанному маршруту

### 6. Канцелярия

Должна уметь:

1. войти под своей ролью
2. открыть карточки документов
3. увидеть маршрут `Входящие и регистрация`
4. увидеть официальный статус документа
5. открыть прямой переход в `NauDoc`, если это предусмотрено ролью

## Сценарии первой недели

### 1. Входящий документ

Должно работать:

1. документ создается
2. получает маршрут регистрации
3. получает официальный статус
4. виден канцелярии

### 2. Внутренний приказ

Должно работать:

1. создается черновик
2. идет согласование
3. идет утверждение
4. после утверждения маршрут не ломается

### 3. Пациентский / клинический документ

Должно работать:

1. врач готовит документ
2. заведующий утверждает
3. канцелярский контур видит документ, если для него обязательна регистрация

### 4. Новый пользователь

Должно работать:

1. администратор создает нового пользователя вручную
2. или пользователь загружается через CSV
3. пользователь может войти
4. его роль дает только нужную видимость

## Конфликтные и пограничные сценарии

Их надо пройти хотя бы один раз до полноценного rollout:

1. два человека открывают один Word-документ
2. два человека открывают одну Excel-таблицу
3. документ уже в `На регистрации`, а пользователь пытается его еще редактировать
4. документ уже `Зарегистрирован`, а инициатор пытается менять содержимое
5. пользователь деактивирован, но участвует в старом маршруте
6. `Bridge` временно не смог обновить статус и требуется повтор sync
7. документ привязан и к проекту, и к заявке

## Что считать ошибкой в первый день

Это реальные blockers:

1. пользователь не может войти
2. Word не открывается
3. Excel не открывается
4. сохранение делает новый файл вместо обновления существующего
5. карточка документа открывается с ошибкой
6. `Bridge` не отдает overview
7. `NauDoc` недоступен для канцелярии или admin
8. пользователю показывается лишний техслой, который ему не нужен для работы

## Что не считать blocker'ом первого дня

Это можно переносить на первую неделю или второй этап:

1. `Workspace Community`
2. тонкая настройка `DocSpace` комнат
3. красивые alias вместо IP
4. live `Workspace` как обязательный маршрут, если основной контур уже работает

## Практический способ прогона

Для автоматизированной приемки:

```bash
bash ops/day1_week1_acceptance.sh
```

Для глубокой приемки по браузеру:

```bash
bash ops/day1_week1_acceptance.sh --browser firefox
```

Отчеты будут лежать в:

```bash
runtime/monitoring/day1-week1-acceptance
```
