# Готовность БД к production

Дата актуальности: `2026-03-28`

## Что считаем базовым и уже закрепленным

Для hospital-production контур использует распределенное хранение:

1. `Rukovoditel / MariaDB`
   - рабочие записи
   - пользователи
   - заявки
   - проекты
   - задачи
   - карточки документов
   - база документов
   - МТЗ

2. `Bridge / SQLite`
   - интеграционные связи
   - журнал sync-ошибок
   - маппинг статусов
   - маппинг полей
   - каталог пользователей
   - identity sources
   - hospital-role mappings
   - document route definitions

3. `NauDoc / Data.fs`
   - официальный документный контур
   - опубликованные объекты и маршрутизация

Это не одна БД и не должно быть одной БД. Для production это нормальная архитектура.

## Что уже проверяется автоматически

Новый audit:

- [audit_database_integrity.py](/home/lebedeffson/Code/Документооборот/ops/audit_database_integrity.py)

Он проверяет:

1. наличие ключевых сущностей `21/22/23/24/25/26/27`
2. наличие обязательных полей в рабочих сущностях
3. отсутствие сиротских ссылок:
   - заявка -> карточка документа
   - проект -> карточка документа
   - карточка документа -> заявка
   - карточка документа -> проект
   - МТЗ -> проект
4. отсутствие рассинхрона перекрестных ссылок
5. отсутствие дублей, когда одна карточка привязана к нескольким заявкам или проектам
6. обязательность маршрута документа для карточек документов
7. наличие обязательных таблиц `Bridge`
8. отсутствие незакрытых `sync_failures`
9. наличие полного набора `document_route_definitions`

Audit включен в общий verify:

- [run_full_verification.sh](/home/lebedeffson/Code/Документооборот/ops/run_full_verification.sh)

## Что было исправлено для целостности данных

1. Автоматически создаваемые карточки документов теперь сразу получают `Маршрут документа`.
2. Для уже созданных карточек без маршрута backfill проходит через обычные sync-скрипты.
3. После исправления в текущем контуре:
   - карточек документов без маршрута: `0`
   - сиротских ссылок между проектами, заявками и карточками: `0`
   - дублирующих привязок одной карточки к нескольким заявкам/проектам: `0`

## Что можно считать уже “замороженным основанием”

Можно считать стабильными:

1. состав основных рабочих сущностей
2. базовую схему их связей
3. наличие document routes как обязательной части карточки документа
4. распределенную схему хранения `Rukovoditel + Bridge + NauDoc`
5. production-lite цикл:
   - backup
   - restore
   - integrity audit
   - full verification

## Что нельзя менять хаотично после старта эксплуатации

1. имена и назначение ключевых entity links
2. смысл полей:
   - `Ссылка на NauDoc`
   - `Связанная карточка документа`
   - `Связанный проект`
   - `Связанная заявка`
   - `Маршрут документа`
3. логику создания карточки документа из проекта/заявки без миграционного сценария
4. структуру `Bridge` таблиц без отдельного migration plan

## Рекомендуемый режим перед каждым релизом

1. `bash ops/check_stack.sh`
2. `python3 ops/audit_database_integrity.py`
3. `python3 ops/audit_ecosystem_integration.py`
4. `bash ops/run_full_verification.sh`

Если `audit_database_integrity.py` не зеленый, релиз или перенос данных дальше делать нельзя.
