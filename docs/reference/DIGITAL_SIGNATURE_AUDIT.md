## Что реально работает

### Администрирование ЭЦП

Доступна страница настройки:

- `https://localhost:18443/docs/manage_signature_form`

На ней отдаются:

1. выбор криптопровайдера
2. выбор хэш-алгоритма
3. выбор длины ключа
4. настройка подписи вложений

### Пользовательский слой

На странице личных данных:

- `https://localhost:18443/docs/personalize_form`

доступны:

1. поле `Сертификат`
2. клиентский код для работы с `CAPICOM.Store`
3. выбор текущего сертификата пользователя

### Документы

Для `HTMLDocument` вкладка `Подписи` реально появляется.

Стендовый документ для проверки:

- `https://localhost:18443/docs/storage/members/admin/digital-signature-smoke/document_signatures_form`

На форме присутствуют:

1. клиентский код `CAPICOM.SignedData`
2. кнопка `Подписать`
3. текст о невозможности подписания без выбранного сертификата
4. загрузка контента документа для подписи

### Ресурсы клиента

Через gateway отдаются:

1. `xenrlinf.cab`
2. `capicom.cab`
3. `certificate.gif`

## Что важно понимать

Полный `end-to-end` сценарий подписи из текущей Linux-среды не может быть честно завершен, потому что legacy-контур ЭЦП завязан на:

1. `ActiveX`
2. `CAPICOM`
3. `xEnroll`
4. локальное хранилище сертификатов Windows

То есть с серверной стороны и по HTML/JS-контролам контур сейчас готов, но для настоящей подписи нужен совместимый клиентский стенд.
