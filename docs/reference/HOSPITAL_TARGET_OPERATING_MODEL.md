# Целевая operating-модель платформы для больницы

Дата: `2026-03-27`

## 1. Главный принцип

Для сотрудников это должна быть **одна платформа**, а не набор отдельных сайтов.

Пользователь не должен думать:

1. где `Rukovoditel`
2. где `NauDoc`
3. где редактор
4. где интеграция

Он должен видеть:

1. один адрес
2. один вход
3. один рабочий кабинет
4. понятные роли и маршруты

Внутри система может состоять из нескольких сервисов, но **снаружи это один продукт**.

---

## 2. Что должно остаться в целевой модели

Логически у платформы должно быть **2 пользовательских контура** и **3 технических ядра**.

### 2.1 Пользовательские контуры

#### `Admin`

Это контур сопровождения и настройки платформы.

Администратор должен уметь:

1. управлять пользователями
2. управлять ролями и группами
3. менять связи и маппинг
4. настраивать шаблоны и маршруты
5. видеть ошибки интеграции
6. запускать retry и ручную перевязку
7. управлять справочниками
8. управлять backup/restore и health-check

#### `User`

Это контур ежедневной работы сотрудников.

Пользователь должен уметь:

1. создавать и редактировать документы
2. работать с заявками
3. работать с карточками документов
4. видеть свои или разрешенные ему записи
5. открывать документ в редакторе
6. отслеживать статус документа
7. работать без технического шума

Пользовательский контур не должен показывать:

1. технастройки
2. системные логи
3. интеграционные детали
4. лишние админские разделы

---

## 3. Технические ядра платформы

### 3.1 `Rukovoditel` — ядро рабочей логики

Это основной рабочий кабинет.

Именно здесь должны жить:

1. пользователи и роли рабочего контура
2. заявки
3. задачи
4. проекты
5. карточки документов
6. база документов
7. МТЗ
8. рабочие отчеты
9. права доступа по ролям

**Вывод:** `Rukovoditel` в платформе нужен обязательно.

Без него не будет:

1. понятной бизнес-логики
2. управления доступом
3. удобного GUI для рабочих процессов

### 3.2 `ONLYOFFICE Docs` — редактор

Это не отдельная бизнес-система, а встроенный редактор.

Его роль:

1. открывать черновик документа
2. давать совместное редактирование
3. не хранить бизнес-логику
4. не быть отдельной точкой входа для пользователей

**Вывод:** `ONLYOFFICE Docs` нужен, но только как редакторный слой.

Пользователь должен попадать в него:

1. из карточки документа
2. без необходимости понимать внутреннюю архитектуру

### 3.3 `NauDoc` — официальный документный контур

Вот здесь нужен честный ответ.

`NauDoc` имеет смысл оставлять только если он действительно нужен как:

1. официальный контур регистрации
2. архив документов
3. система номенклатуры и хранения
4. контур официального статуса документа
5. legacy-СЭД, которую нельзя просто убрать

Если этих функций реально требуют:

1. регламент учреждения
2. архивная политика
3. внутренний документооборот
4. сохранение совместимости с существующей практикой

тогда `NauDoc` нужно оставлять.

Но тогда его надо делать **невидимым как отдельный продукт**:

1. пользователь работает в рабочем кабинете
2. официальный статус живет в `NauDoc`
3. переход в `NauDoc` нужен только там, где это действительно необходимо

Если же `NauDoc` дублирует то, что уже можно делать в `Rukovoditel`, и не добавляет уникальной ценности, тогда в долгосрочной перспективе его лучше вытеснять из пользовательского сценария.

**Практический вывод:**

1. для первой production-волны `NauDoc` лучше оставить
2. но сделать его служебным официальным контуром
3. не строить на нем основной UI для сотрудников

### 3.4 `Bridge` — не отдельная часть платформы для пользователя

`Bridge` нужен, но он не должен восприниматься как четвертая пользовательская система.

Его правильная роль:

1. хранить связи
2. держать статусы интеграции
3. показывать ошибки синхронизации
4. давать retry и manual relink
5. быть admin-центром интеграции

**Вывод:** `Bridge` нужен как внутренний слой, но не как отдельная рабочая часть для сотрудников.

## 4. Нужно ли оставлять все реально

### Оставить обязательно

1. `Rukovoditel`
2. `ONLYOFFICE Docs`

### Оставить условно, но скрыть вглубь

1. `NauDoc`
2. `Bridge`

### Включать в первую hospital production-волну ограниченно

1. `ONLYOFFICE DocSpace`
2. `ONLYOFFICE Workspace`

Их нужно включать только в ограниченном scope первой волны:

1. `DocSpace` — как внешний collaboration-контур:
   - `Collaboration rooms`
   - `Public rooms`
   - при необходимости `Form filling rooms`
2. `Workspace` — как дополнительный легкий корпоративный сервисный слой:
   - `Calendar`
   - опционально `Community`

При этом:

1. они не должны становиться основным пользовательским кабинетом
2. они не должны дублировать процессы `Rukovoditel`
3. до релиза первой волны их нужно реально развернуть и проверить
4. `Workspace Mail`, `CRM`, `Projects` и `Documents` не должны входить в первую волну

## 5. Как должна выглядеть единая модель пользователя

Сейчас одна из главных проблем — это разные учетные контуры.

Для hospital production нужна модель:

1. **один пользователь**
2. один логин
3. одна роль
4. одна привязка к подразделению
5. единая карта соответствия между системами

### Целевая схема

1. один источник истины по пользователям
2. `LDAP/AD` или единый внутренний каталог
3. один `username / employee_id`
4. одинаковая роль в `Rukovoditel` и `NauDoc`

### Переходный вариант

До `LDAP/SSO` можно жить так:

1. `Rukovoditel` — основной каталог пользователей
2. `Bridge` — реестр соответствий
3. `NauDoc` — привязан к тем же `username`

Это уже позволяет идти к модели “один пользователь”, не ломая всю платформу сразу.

## 6. Как должны быть разделены полномочия

### `Admin`

Должен видеть и уметь:

1. пользователи
2. группы пользователей
3. роли
4. шаблоны
5. маршруты документов
6. справочники
7. интеграционные связи
8. журнал sync-ошибок
9. backup/restore
10. health и monitoring

### `User`

Должен видеть только рабочую часть.

#### Базовая логика доступа

1. пользователь видит свои документы
2. пользователь видит документы, где он ответственный
3. пользователь видит документы своего подразделения, если это разрешено ролью
4. заведующий видит контур подразделения
5. канцелярия видит документы, идущие через регистрацию и архив
6. администратор видит всё

### Что лучше не делать

Не стоит сразу давать обычному пользователю полный обзор “чужих” документов.

Логичнее базовая модель:

1. свой контур
2. назначенные записи
3. отделенческий контур для руководителя
4. расширенный контур только для канцелярии и админа

Это и безопаснее, и понятнее для больницы.

## 7. Как должна быть разделена ответственность между системами

### `Rukovoditel`

Отвечает за:

1. карточку процесса
2. заявку
3. проект
4. рабочую карточку документа
5. роли и права
6. маршрутизацию внутри рабочего контура

### `ONLYOFFICE Docs`

Отвечает за:

1. редактирование файла
2. совместную работу над черновиком

### `NauDoc`

Отвечает за:

1. официальный документ
2. регистрационный номер
3. архив
4. официальный статус документа
5. хранение в документном контуре

### `Bridge`

Отвечает за:

1. связь сущностей
2. журнал синхронизации
3. retry
4. relink
5. маппинг статусов и полей

## 8. Самая логичная целевая схема для больницы

### Для пользователя

Один адрес:

1. вход в платформу
2. рабочие модули
3. карточка документа
4. кнопка `Открыть в редакторе`
5. при необходимости кнопка `Открыть официальный контур`

Пользователь не должен сам выбирать:

1. в какой системе он сейчас работает
2. где лежит документ
3. куда сохраняется статус

### Для администратора

Отдельный admin-контур внутри той же платформы:

1. пользователи
2. роли
3. маршруты
4. шаблоны
5. связи
6. интеграции
7. backup/restore

### Для архитектуры

1. `Rukovoditel` — front-door и ядро рабочих процессов
2. `ONLYOFFICE Docs` — встроенный редактор
3. `NauDoc` — официальный документный контур
4. `Bridge` — внутренний интеграционный слой

---

## 9. Практическое решение, которое лучше всего делать дальше

### Оставляем

1. `Rukovoditel`
2. `ONLYOFFICE Docs`
3. `NauDoc`
4. `Bridge`

### Но меняем логику показа

1. пользователю показываем только единую платформу
2. `NauDoc` не торчит как отдельный обязательный сайт
3. `Bridge` остается только для admin
4. `ONLYOFFICE` открывается только из документа

### Следующий правильный порядок

1. production `.env` и боевые секреты
2. единый каталог пользователей / `LDAP-first`
3. финальная hospital role matrix
4. финальные hospital document routes
5. прямой `Bridge -> NauDoc` write-back
6. monitoring, staging, pilot

---

## 10. Короткий управленческий вывод

Если смотреть объективно, все четыре технических компонента сейчас выполняют разные функции:

1. `Rukovoditel` — рабочая логика
2. `ONLYOFFICE Docs` — редактор
3. `NauDoc` — официальный документный контур
4. `Bridge` — синхронизация и контроль

Но для сотрудников это не должны быть четыре отдельные системы.

Правильная целевая модель такая:

1. **одна платформа**
2. **два режима**: `admin` и `user`
3. **три видимых слоя смысла**:
   - рабочая логика
   - документы
   - редактор
4. все остальное — внутренний механизм платформы
