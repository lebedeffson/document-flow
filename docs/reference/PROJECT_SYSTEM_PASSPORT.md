# Паспорт проекта: hospital production платформа документооборота

Дата актуальности: `2026-03-30`

## 1. Что это за проект

Это единая hospital-платформа документооборота, в которой пользователь должен видеть не набор разрозненных систем, а один рабочий продукт.

Платформа закрывает:

1. рабочие карточки документов
2. заявки и сервисные обращения
3. задачи и поручения
4. проекты и инициативы
5. документный маршрут от черновика до официального статуса
6. встроенное веб-редактирование документов и таблиц
7. служебную интеграцию с официальным контуром документа

## 2. Как устроена платформа

### 2.1 Внешне

Для пользователя это один адрес и один кабинет:

1. одна точка входа
2. одна авторизация
3. понятные роли
4. понятные разделы

### 2.2 Внутри

Платформа состоит из нескольких сервисов:

1. `Rukovoditel`
   Это основной рабочий кабинет и ядро бизнес-логики.
2. `ONLYOFFICE Docs`
   Это встроенный редактор Word/Excel/документных файлов.
3. `NauDoc`
   Это официальный служебный контур регистрации и хранения документа.
4. `Bridge`
   Это интеграционный слой между рабочей карточкой и официальным контуром.
5. `ONLYOFFICE DocSpace`
   Это дополнительный collaboration-контур первой волны.
6. `ONLYOFFICE Workspace`
   Это дополнительный корпоративный сервисный слой первой волны.

## 3. Что является главным контуром

Главный рабочий путь первой production-волны такой:

1. пользователь работает в `Rukovoditel`
2. документ открывается в `ONLYOFFICE Docs`
3. официальный статус живет в `NauDoc`
4. связь обслуживает `Bridge`

`DocSpace` и `Workspace` не заменяют основной кабинет.

## 4. Какие модули уже входят в продукт

### Рабочие модули

1. `Проекты и инициативы`
2. `Поручения и задачи`
3. `Заявки на обслуживание`
4. `Карточки документов`
5. `База документов`
6. `Заявки на МТЗ`
7. контрольные и маршрутные отчеты

### Официальный контур

1. регистрация документа
2. архивный и служебный контур
3. справочники и служебные разделы `NauDoc`

### Office-контур первой волны

`DocSpace`:

1. `Collaboration rooms`
2. `Public rooms`
3. при необходимости `Form filling rooms`

`Workspace`:

1. `Calendar`
2. опционально `Community`

Что сознательно не входит в первую волну:

1. `Workspace Mail`
2. `Workspace CRM`
3. `Workspace Projects`
4. `Workspace Documents`

## 5. Роли в системе

### `admin`

Полный контур сопровождения:

1. пользователи и роли
2. настройки
3. резервные копии
4. логи
5. интеграции
6. контроль здоровья контура

### `head`

Заведующий отделением / руководитель:

1. контроль
2. согласование и утверждение
3. доступ к рабочим и контрольным отчетам

### `doctor`

Основной рабочий пользователь:

1. документы
2. заявки
3. проекты
4. задачи
5. редактор документов

### `nurse`

Координатор отделения:

1. документы отделения
2. маршруты
3. рабочие таблицы
4. сопровождение процессов без системных прав

### `registry`

Регистратура / заявитель:

1. создает обращения
2. видит свои и связанные записи
3. не получает лишний доступ

### `office`

Канцелярия:

1. регистрация документов
2. контроль официального маршрута
3. работа с карточками и статусами

## 6. Боевые маршруты первой волны

Зафиксированы три основных маршрута:

1. `Входящие и регистрация`
2. `Внутренние приказы`
3. `Пациентские и клинические документы`

Единый словарь статусов:

1. `Черновик`
2. `На согласовании`
3. `На утверждении`
4. `Подписан`
5. `На ознакомлении`
6. `На регистрации`
7. `Зарегистрирован`
8. `Архивирован`

Где что является источником истины:

1. `Rukovoditel` — рабочая карточка и бизнес-процесс
2. `Bridge` — связь и интеграционный статус
3. `NauDoc` — официальный объект документа

## 7. Что пользователь делает в системе

Основной сценарий:

1. входит в систему
2. открывает карточку документа
3. открывает документ в `ONLYOFFICE`
4. вносит правки
5. сохраняет изменения в тот же файл
6. видит статус документа
7. при необходимости документ попадает в официальный контур

Важно:

1. пользователь не должен вручную скачивать и снова загружать файл
2. документ сохраняется в существующую запись
3. `NauDoc` не должен быть главным UI для обычного пользователя

## 8. Точки входа

В closed-LAN и same-host режиме используются такие адреса:

1. платформа: `https://<server-ip>/`
2. официальный контур: `https://<server-ip>/docs/`
3. интеграционный контур: `https://<server-ip>/bridge/`
4. редактор: `https://<server-ip>/office/`
5. `DocSpace`: `https://<server-ip>/docspace/`
6. `Workspace`: `https://<server-ip>/workspace/`

## 9. Как ставится проект

Есть два основных режима установки.

### 9.1 Portable bundle

Для закрытой сети и выезда с флешкой:

```bash
bash install_everything.sh /opt/docflow
```

### 9.2 Установка из git

Если на сервере есть интернет:

```bash
bash install_from_git.sh
```

### 9.3 One-shot live office

Если сервер тянет live office слой:

```bash
sudo bash install_everything.sh --with-live-office --office-auto-host /opt/docflow
```

## 12. Что нужно понимать про сервер

### Для логики продукта

По механике проекта платформа уже production-ready.

### Для ресурсов

При полном same-host профиле:

1. `DocFlow`
2. `ONLYOFFICE Docs`
3. `DocSpace`
4. `Workspace`

сервер с `16 GB RAM` — это пограничный режим.

Практический вывод:

1. для показа и аккуратного первого запуска этого может хватить
2. для спокойного hospital-prod лучше `32 GB RAM`
3. первую неделю стоит мониторить память, swap и health

## 13. Что считать готовностью к пилоту

Контур готов к пилоту, если одновременно:

1. стек поднимается без ручных костылей
2. все ключевые audit-скрипты зеленые
3. Word и Excel сохраняются в существующие файлы
4. роли и видимость подтверждены
5. `Bridge` и `NauDoc` живы
6. сценарная приемка первого дня проходит
