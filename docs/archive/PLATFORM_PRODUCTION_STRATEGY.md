# Стратегия развития платформы до полноценной production-версии

Дата: `2026-03-24`

## 1. Зачем этот документ

Этот документ фиксирует:

1. в каком состоянии находится текущая платформа
2. чего не хватает до полноценной рабочей версии
3. какие есть варианты дальнейшего развития
4. стоит ли сейчас переписывать стек
5. какой путь для нас наиболее реалистичен

Документ опирается на уже собранный стенд:

1. `NauDoc` как legacy-контур документов
2. `Rukovoditel` как рабочий пользовательский слой
3. `middleware` как интеграционный слой

---

## 2. Короткий вывод

На текущем этапе **не стоит сразу переписывать всю систему с нуля**.

Наиболее рациональный путь:

1. довести текущую гибридную платформу до пилотной и затем production-lite версии
2. вынести в современный стек только те части, где legacy реально мешает:
   - единый портал
   - интеграция
   - уведомления
   - формы подачи
   - аналитика
3. к полному переписыванию ядра документооборота возвращаться только после пилота и стабилизации процессов

Иначе есть риск потратить много месяцев на перенос уже работающей логики вместо запуска полезной системы для пользователей.

---

## 3. Текущее состояние платформы

### 3.1 Что уже есть

Сейчас у нас уже не просто “набор идей”, а рабочий каркас системы:

1. поднят гибридный стенд `NauDoc + Rukovoditel + middleware`
2. в `Rukovoditel` собраны основные рабочие сущности:
   - проекты
   - задачи и поручения
   - заявки на обслуживание
   - карточки документов
   - база документов
   - заявки на МТЗ
3. настроены роли, статусы, поля, связи и базовые dashboard-отчеты
4. работает сценарий:
   - заявка -> карточка документа -> bridge -> обратная синхронизация
5. работает сценарий:
   - проект -> карточка документа -> bridge -> обратная синхронизация
6. есть background worker для периодической синхронизации
7. в интерфейсе видны статусы документа / интеграции
8. `middleware` умеет хранить связи, фильтровать проблемные записи и возвращать данные обратно в рабочий контур

### 3.2 Что это значит по зрелости

Текущая зрелость решения:

1. как технический стенд: высокая
2. как пилотный продукт: средняя
3. как production-система: пока недостаточная

Причина не в отсутствии идеи, а в том, что до production не хватает не только функций, но и эксплуатационного слоя:

1. безопасности
2. отказоустойчивости
3. регламентов
4. трассировки ошибок
5. единого UX
6. формализованных маршрутов документов

---

## 4. Что значит “полноценная реальная версия”

Чтобы считать систему полноценной production-платформой документооборота, нужно закрыть 4 контура одновременно.

### 4.1 Функциональный контур

Должны быть полноценно покрыты:

1. электронная канцелярия:
   - карточка документа
   - файл документа
   - согласование
   - утверждение
   - подписание
   - ознакомление
   - регистрация
   - архивирование
2. проекты и поручения
3. заявки на обслуживание
4. база документов
5. обсуждения / коммуникации по карточке
6. ЭЦП
7. МТЗ
8. настраиваемость полей, форм, статусов и маршрутов

### 4.2 Пользовательский контур

Пользователь должен получать:

1. единый понятный вход
2. работу через браузер
3. формы без перегрузки
4. понятные роли и экраны
5. минимум ручного перехода между системами
6. прямые ссылки на документы и связанные процессы

### 4.3 Серверный и эксплуатационный контур

Должны быть:

1. единый reverse proxy
2. `HTTPS`
3. staging и production
4. резервное копирование
5. мониторинг
6. централизованные логи
7. регламент восстановления
8. обновляемая конфигурация

### 4.4 Контур управления и внедрения

Должны быть:

1. матрица ролей и прав
2. регламент маршрутов
3. инструкции сотрудника, руководителя и администратора
4. тестовые сценарии приемки
5. пилотная группа
6. план сопровождения после запуска

## 5. Что нужно доделать до production

Ниже список не “красивых пожеланий”, а обязательных блоков.

### 5.1 Документный контур

Нужно:

1. формализовать типы документов:
   - входящий
   - исходящий
   - внутренний
   - служебная записка
   - приказ
   - договор
2. для каждого типа описать:
   - карточку
   - маршрут
   - ответственных
   - переходы статусов
   - правила завершения
3. унифицировать словарь статусов между `Rukovoditel`, `middleware` и `NauDoc`
4. закрыть сценарий регистрации и архивирования без ручной потери статуса

### 5.2 Рабочий контур сотрудников

Нужно:

1. дочистить карточки сущностей и формы
2. убрать лишние поля из пользовательских представлений
3. сделать отдельные views:
   - сотрудник
   - руководитель
   - канцелярия
4. добавить быстрые действия:
   - создать заявку
   - создать поручение
   - создать карточку документа
5. сделать понятные списки:
   - мои документы
   - мои задачи
   - мои заявки
   - мои согласования

### 5.3 Руководительский контур

Нужно:

1. панели контроля по срокам
2. просрочки и риски
3. документы на согласовании
4. заявки без завершения
5. проекты с риском по срокам и документам
6. сводки по подразделениям

### 5.4 Интеграционный контур

Нужно:

1. завершить таблицу соответствия статусов
2. сделать устойчивую очередь синхронизации
3. добавить retry-механику для ошибочных связей
4. сделать нормальный журнал интеграционных событий
5. добавить технические идентификаторы корреляции
6. подготовить входные каналы:
   - web-form
   - email-to-ticket
   - потом при необходимости bot / messenger

### 5.5 Безопасность и доступ

Нужно:

1. привести роли и права к единой матрице
2. закрыть админские сервисы от прямого внешнего доступа
3. включить `HTTPS`
4. нормализовать хранение секретов и паролей
5. подготовить `SSO/LDAP` как отдельный этап
6. добавить аудит действий по ключевым операциям

### 5.6 Эксплуатация

Нужно:

1. собрать единый внешний вход
2. разделить `staging` и `production`
3. настроить backup:
   - `MariaDB`
   - `bridge.db`
   - данные `NauDoc`
4. настроить мониторинг:
   - доступность
   - синхронизация
   - ошибки worker
   - заполнение диска
5. описать процедуру обновления
6. описать процедуру восстановления после сбоя

### 5.7 Внедрение и обучение

Нужно:

1. короткая инструкция сотрудника
2. короткая инструкция руководителя
3. инструкция администратора
4. пилот на 1-2 подразделениях
5. журнал замечаний
6. цикл доработок после пилота

## 6. Главный вопрос по стеку

### 6.1 Что у нас сейчас

Текущая схема:

1. `NauDoc` на очень старом legacy-стеке
2. `Rukovoditel` на живом, но не “идеальном” PHP-стеке
3. `middleware` как наш собственный слой интеграции

### 6.2 Где стек реально болит

Больные места сейчас такие:

1. `NauDoc` неудобен как главный UI
2. в `NauDoc` рискованно делать большой объем новой логики
3. пользовательский опыт в гибридной схеме еще не выглядит как единый продукт
4. серверный контур пока тестовый, а не production-grade

### 6.3 Где стек пока не является критической проблемой

Стек пока не мешает нам:

1. собрать пилот
2. запустить реальную рабочую схему для ограниченной группы
3. подтвердить нужные пользователям процессы
4. понять, какие функции действительно должны жить в новом контуре, а какие можно оставить

## 7. Варианты дальнейшего развития

## Вариант A. Довести текущую гибридную платформу без большого rewrite

### Суть

Оставляем:

1. `NauDoc` как документное ядро
2. `Rukovoditel` как пользовательскую оболочку
3. `middleware` как точку интеграции

Дорабатываем:

1. UX
2. роли и формы
3. маршруты
4. синхронизацию
5. входные каналы
6. серверную эксплуатацию

### Что получим

1. самую быструю дорогу к production-lite
2. минимальный риск потери уже работающей логики
3. возможность начать пилот и получать обратную связь

### Плюсы

1. быстрый результат
2. низкий риск
3. минимальный объем переписывания
4. можно внедрять поэтапно

### Минусы

1. остается зависимость от legacy `NauDoc`
2. архитектура остается составной
3. UX придется “сшивать”, а не получать из коробки

### Когда выбирать

1. если нам важен быстрый запуск
2. если бюджет ограничен
3. если нужно сначала подтвердить процессы на реальных пользователях

### Оценка

1. до pilot-ready: `6-10 недель`
2. до production-lite: `3-5 месяцев`

---

## Вариант B. Частичная модернизация с новым внешним слоем

### Суть

Оставляем `NauDoc` только как специализированный legacy-контур документов, а вокруг него строим современный внешний слой.

Практически это означает:

1. `middleware` превращаем в полноценный integration/API слой
2. постепенно выносим критичный UX из `Rukovoditel` в новый портал
3. новые функции строим уже не в legacy, а на новом сервисном слое

### Что можно вынести в новый стек в первую очередь

1. единый портал и личный кабинет
2. руководительские панели
3. web-формы заявок
4. уведомления
5. интеграционные очереди
6. единый API
7. аудит и журналы операций

### Рекомендуемый современный стек для такого пути

1. backend:
   - `Python 3.12`
   - `Django` или `FastAPI`
2. frontend:
   - `React + TypeScript`
3. DB:
   - `PostgreSQL`
4. async / jobs:
   - `Celery + Redis` или `RabbitMQ`
5. auth:
   - `Keycloak` или `LDAP/SSO` поверх reverse proxy
6. files:
   - `MinIO/S3`
7. reverse proxy:
   - `Nginx` или `Traefik`

### Что получим

1. современный UX
2. постепенный выход из legacy
3. сохранение уже работающего документного ядра
4. возможность мигрировать без “большого взрыва”

### Плюсы

1. хороший баланс между скоростью и качеством
2. можно не ломать текущие процессы
3. новые модули делаются уже на современном стеке

### Минусы

1. архитектура на время станет еще сложнее
2. придется поддерживать и старый, и новый контур
3. нужен более сильный архитектурный контроль

### Когда выбирать

1. если пилот подтвердил ценность платформы
2. если есть бюджет на развитие минимум на 2 этапа
3. если UX и интеграции уже становятся главным ограничением

### Оценка

1. production на гибриде + современный портал: `5-9 месяцев`
2. с поэтапным вводом модулей: `6-12 месяцев`

## Вариант C. Полный rewrite всей платформы

### Суть

Переносим в новый стек:

1. документное ядро
2. рабочие сущности
3. маршруты
4. архив
5. интеграции
6. панели
7. права и аудит

### Что это реально значит

Это не просто “переписать UI”.

Это означает заново реализовать:

1. документные карточки
2. версии
3. согласование
4. регистрационные журналы
5. архивную логику
6. права доступа
7. ЭЦП-контур
8. миграцию исторических данных

### Плюсы

1. чистая современная архитектура
2. меньше legacy-долга в будущем
3. больше свободы по UX и интеграциям

### Минусы

1. самый длинный путь
2. самый высокий риск
3. высокая цена ошибки
4. можно потерять важные нюансы текущей логики документооборота

### Когда выбирать

1. если гибрид уже доказал процессы
2. если понятно, что legacy нельзя оставить даже как документный контур
3. если подтверждены бюджет, команда и сроки

### Оценка

1. MVP: `6-9 месяцев`
2. функциональный паритет и внедрение: `12-18 месяцев`
3. с миграцией истории и ЭЦП: часто `12+ месяцев`

## 9. Рекомендуемый путь

Рекомендую идти в 3 фазы.

## Фаза 1. Довести гибрид до production-lite

Срок: `0-3 месяца`

Цель:

1. единый рабочий интерфейс
2. формальные маршруты документов
3. устойчивая синхронизация
4. базовая эксплуатация
5. пилот на реальных пользователях

Ключевые задачи:

1. reverse proxy + единый URL + `HTTPS`
2. backup и monitoring
3. журнал интеграционных ошибок
4. email/web-form входы
5. руководительские панели
6. инструкции и пилот

## Фаза 2. Частичная модернизация

Срок: `3-9 месяцев`

Цель:

1. убрать самые болезненные UX-ограничения
2. вынести новые модули на современный стек
3. оставить `NauDoc` только там, где он действительно нужен

Ключевые задачи:

1. новый портал или modern UI слой
2. единый API и очередь событий
3. аудит и журнал операций
4. web-кабинеты по ролям
5. `SSO/LDAP`

## Фаза 3. Решение о rewrite ядра

Срок: после пилота и после фазы 2

Решение принимаем только если одновременно выполняются условия:

1. процессы уже устоялись
2. пилот показал реальные узкие места legacy
3. есть утвержденный бюджет
4. есть команда под системный rewrite

## 10. Что делать прямо сейчас

Если идти прагматично, следующий пакет работ должен быть таким.

### Блок 1. Production-hardening

1. единый reverse proxy
2. единый внешний URL
3. `HTTPS`
4. backup `MariaDB`, `bridge.db`, данных `NauDoc`
5. мониторинг и healthchecks
6. централизованные логи

### Блок 2. Завершение функционального каркаса

1. финализировать типы документов
2. описать маршруты
3. привести статусы к единому словарю
4. довести карточки и формы по ролям
5. доделать руководительские панели

### Блок 3. Завершение интеграции

1. retry-механизм
2. журнал ошибок
3. email/web-form вход
4. автосинхронизация без ручных шагов
5. единые ссылки на документ и его статус

### Блок 4. Подготовка к пилоту

1. инструкции
2. приемочные сценарии
3. пилотная группа
4. сбор замечаний
5. цикл доработок

## 11. Итоговая рекомендация

Если цель состоит в том, чтобы **получить полноценную работающую платформу в разумный срок**, то лучший путь сейчас такой:

1. **не переписывать всю систему немедленно**
2. **довести текущий гибрид до production-lite**
3. **после пилота вынести современный внешний слой и интеграционные сервисы**
4. **переписывать ядро документооборота только после того, как это станет технически и экономически оправдано**

Иными словами:

1. быстрый путь к полезной системе: `гибрид`
2. лучший путь к долгосрочной архитектуре: `гибрид -> частичная модернизация -> при необходимости rewrite`

Это даст нам:

1. быстрый запуск
2. сохранение уже работающей логики
3. меньше риск потерять документный контур
4. более здравую стоимость развития
