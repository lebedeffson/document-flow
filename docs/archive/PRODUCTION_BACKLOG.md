# Production Backlog

Дата: `2026-03-24`

## 1. Цель

Довести текущую платформу `NauDoc + Rukovoditel + middleware` до состояния `production-lite`, в котором:

1. пользователи работают через понятный веб-интерфейс
2. документы, заявки и проекты связаны между собой
3. серверная часть работает устойчиво и предсказуемо
4. систему можно запускать на реальной пилотной группе

## 2. Принцип работы

Роли контуров остаются такими:

1. `NauDoc`:
   - документ
   - версия
   - согласование
   - архив
   - ЭЦП-контур
2. `Rukovoditel`:
   - ежедневная работа пользователей
   - проекты
   - заявки
   - задачи
   - карточки документов
   - МТЗ
3. `middleware`:
   - связка записей
   - статусы
   - журнал интеграции
   - входные каналы

## 3. Что считаем релизом production-lite

Релиз считается готовым, если одновременно выполнено:

1. есть единый внешний вход и `HTTPS`
2. есть backup и базовый мониторинг
3. проекты, заявки и карточки документов связаны автоматически
4. статусы документов видны в рабочем интерфейсе
5. описаны и работают типовые маршруты документов
6. есть панели для сотрудников и руководителей
7. есть инструкции и пилотная группа

---

## 4. Приоритеты

### `P0` Критично до пилота

1. единый reverse proxy и единый URL
2. `HTTPS`
3. backup всех контуров
4. журнал интеграционных ошибок
5. retry по проблемным связям
6. словарь статусов и маршрутов документов
7. роли и права
8. руководительские панели
9. инструкции и чеклисты запуска

### `P1` Очень желательно в production-lite

1. web-форма подачи заявок
2. `email -> заявка`
3. аудит ключевых действий
4. staging-контур
5. уведомления по email

### `P2` Можно после пилота

1. `SSO/LDAP`
2. современный единый портал поверх текущего контура
3. расширенный realtime-чат
4. более глубокая аналитика
5. частичный вынос новых модулей на новый стек

## 4.1 Рабочие спецификации

Для `P0` уже подготовлены опорные документы:

1. `ROLE_MATRIX.md` - матрица ролей и прав
2. `DOCUMENT_ROUTE_MATRIX.md` - словарь статусов и типовые маршруты документов

## 5. Эпики и задачи

## Epic A. Production-инфраструктура

### Цель

Сделать из тестового стенда управляемый серверный контур.

### Задачи

1. Поднять единый reverse proxy:
   - `Nginx` или `Traefik`
   - единая точка входа
   - маршрутизация к `Rukovoditel`, `middleware`, `NauDoc`
2. Настроить `HTTPS`.
3. Закрыть прямой внешний доступ к внутренним сервисам.
4. Подготовить prod-конфиги и отдельные volume.
5. Описать переменные окружения и секреты.

### Acceptance Criteria

1. пользователь работает через один внешний URL
2. внутренние сервисы не торчат наружу напрямую
3. сертификат и `HTTPS` работают штатно
4. запуск воспроизводим одним набором compose-конфигов

## Epic B. Backup и восстановление

### Цель

Исключить ситуацию “все работает, пока ничего не сломалось”.

### Задачи

1. Backup `MariaDB`.
2. Backup `bridge.db`.
3. Backup данных `NauDoc`.
4. Ротация и хранение копий.
5. Скрипт тестового восстановления.
6. Инструкция восстановления после сбоя.

### Acceptance Criteria

1. резервные копии создаются по расписанию
2. копии лежат вне рабочего контейнера
3. есть проверенный сценарий восстановления

## Epic C. Документные маршруты

### Цель

Перевести документооборот из “технически возможно” в “формально управляется”.

### Задачи

1. Утвердить типы документов:
   - входящий
   - исходящий
   - внутренний
   - служебная записка
   - приказ
   - договор
2. Для каждого типа описать:
   - карточку
   - обязательные поля
   - маршрут
   - ответственных
   - статусы
   - условие завершения
3. Свести статусы между `Rukovoditel`, `middleware` и `NauDoc`.
4. Подготовить матрицу переходов статусов.
5. Зафиксировать регламент регистрации и архивирования.

### Acceptance Criteria

1. по каждому типу документа есть утвержденный маршрут
2. статусы не конфликтуют между системами
3. сотрудник и руководитель понимают, на каком этапе документ

## Epic D. Пользовательский UX

### Цель

Сделать систему удобной для сотрудников и руководителей.

### Задачи

1. Очистить карточки от лишних полей.
2. Разделить представления по ролям:
   - сотрудник
   - руководитель
   - канцелярия
   - администратор
3. Подготовить быстрые экраны:
   - мои задачи
   - мои документы
   - мои заявки
   - мои согласования
4. Упростить создание:
   - заявки
   - поручения
   - карточки документа
5. Сделать видимыми:
   - статус документа
   - наличие связанного документа
   - риск просрочки

### Acceptance Criteria

1. пользователь видит только нужные ему действия
2. карточки не перегружены
3. статус документа виден без перехода в другую систему

## Epic E. Руководительские панели

### Цель

Дать управленческую картину, а не просто список записей.

### Задачи

1. Панель:
   - документы на согласовании
   - документы на утверждении
   - просроченные задачи
   - заявки в риске
   - проекты с риском по документам
2. Фильтры по подразделениям.
3. Сводки по исполнителям.
4. Сводки по срокам и просрочкам.

### Acceptance Criteria

1. руководитель видит картину за 1 экран
2. можно быстро найти проблемные процессы

## Epic F. Интеграционная надежность

### Цель

Сделать интеграцию устойчивой, а не “работающей в удачных случаях”.

### Задачи

1. Завершить таблицу соответствия статусов.
2. Добавить журнал интеграционных событий.
3. Добавить retry для проблемных связей.
4. Добавить причину ошибки и время последней попытки.
5. Добавить correlation id для операций.
6. Подготовить отдельный отчет:
   - связи с ошибкой
   - связи без финального документа
   - зависшие связи

### Acceptance Criteria

1. каждая ошибка синхронизации видна
2. проблемную связь можно переобработать
3. есть понимание, где именно сломался сценарий

## Epic G. Входные каналы заявок

### Цель

Снизить ручной ввод и приблизить систему к реальной эксплуатации.

### Задачи

1. Web-форма подачи заявок.
2. `email -> заявка`.
3. Правила категоризации и назначения исполнителя.
4. Шаблоны уведомлений по созданию и закрытию.

### Acceptance Criteria

1. пользователь может создать заявку без входа в сложные сущности
2. базовый поток заявок приходит в систему автоматически

## Epic H. Безопасность и аудит

### Цель

Подготовить систему к реальному использованию без явных организационных дыр.

### Задачи

1. Матрица ролей и прав.
2. Ограничение доступа к админским разделам.
3. Смена тестовых паролей и вынос секретов.
4. Журнал ключевых действий:
   - создание документа
   - смена статуса
   - согласование
   - завершение
5. Подготовка к `SSO/LDAP` как следующему этапу.

### Acceptance Criteria

1. права разграничены по ролям
2. опасные действия аудируются
3. тестовые креды не остаются в рабочем контуре

## Epic I. Пилот и внедрение

### Цель

Сделать переход от стенда к реальным пользователям управляемым.

### Задачи

1. Подготовить пилотную группу `1-2` подразделения.
2. Подготовить сценарии приемки.
3. Подготовить инструкции:
   - сотрудник
   - руководитель
   - администратор
4. Подготовить форму сбора замечаний.
5. Подготовить цикл исправлений после пилота.

### Acceptance Criteria

1. пилот идет по заранее описанному сценарию
2. обратная связь собирается системно
3. после пилота есть список обязательных доработок

## 7. Что не делаем до production-lite

Чтобы не расползтись, до production-lite сознательно не берем:

1. полный rewrite `NauDoc`
2. большой новый фронтенд с нуля
3. полноценный realtime-chat
4. полный складской ERP вместо базового МТЗ
5. глубокую миграцию всего ядра документооборота на новый стек

---

## 8. Definition of Done для production-lite

Production-lite достигнут, если:

1. пользователь заходит по одной ссылке
2. заявки, проекты и документы связаны без ручного хаоса
3. статус документа виден в рабочем интерфейсе
4. документы идут по типовым маршрутам
5. есть backup, monitoring и регламент восстановления
6. есть руководительские панели
7. есть инструкции и отработанный пилотный сценарий

---

## 9. Следующий практический шаг

Первый рабочий пакет должен быть таким:

1. `reverse proxy + HTTPS + единый URL`
2. `backup + monitoring`
3. матрица ролей и прав
4. словарь статусов и маршрутов документов

Именно этот пакет лучше брать следующим спринтом, потому что он одновременно:

1. приближает нас к production
2. уменьшает технический риск
3. создает базу для последующих бизнес-доработок
