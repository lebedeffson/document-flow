# План заказческого показа платформы

Дата фиксации: `2026-03-26`

## 1. Цель показа

Показать платформу не как набор разрозненных сервисов, а как единую рабочую систему, в которой:

1. сотрудник работает через понятный веб-интерфейс
2. руководитель видит контроль и статусы
3. документы связаны с заявками и проектами
4. документный контур и рабочий контур синхронизированы
5. совместная работа над документом возможна в браузере

## 2. Что уже можно показывать

### 2.1 Единая платформа

Сейчас уже работает единый внешний вход:

1. `https://localhost:18443/` — основной вход
2. `https://localhost:18443/docs/` — документный контур `NauDoc`
3. `https://localhost:18443/bridge/` — интеграционный слой

Для локального стенда важно использовать именно эти адреса.

1. `https://localhost/docs/` сейчас не настроен
2. для показа нужно открывать контур только через `https://localhost:18443/`

Для заказческого показа это можно подавать как:

1. единая платформа с рабочим контуром
2. документное ядро
3. слой синхронизации между процессами и документами

### 2.2 Пользовательский контур

1. проекты и инициативы

3. заявки на обслуживание
4. карточки документов
5. база документов
6. заявки на МТЗ

Уже настроены:

1. меню по ролям
2. базовые статусы
3. dashboard-отчеты
4. рабочие формы
5. связи между проектами, заявками и карточками документов

### 2.3 Интеграция

Уже подтверждены рабочие сценарии:

1. `заявка -> карточка документа -> bridge -> обратная синхронизация`

3. возврат ссылки и статуса документа в рабочий контур

## 4. Что нужно доделать перед показом заказчику

### 4.1 Обязательно

Это минимальный пакет, без которого показ будет выглядеть как технический стенд, а не как продукт.

1. оформить главный экран и dashboard более презентационно
2. скрыть все технические разделы, которые не нужны заказчику
3. подготовить 3-4 фиксированных сценария показа
4. заранее открыть вкладки под роли и контуры

Уже сделано:

1. названия системы и логин-экрана приведены к заказческому виду
2. демо-данные очищены от тестовых названий
3. ключевые записи уже связаны между собой
4. подготовительный seed вынесен в отдельный скрипт

### 4.2 Желательно

Это заметно усилит впечатление.

1. брендировать систему под название проекта или компании
2. сделать более аккуратный стартовый экран для сотрудника
3. сделать отдельный управленческий экран для руководителя
4. показать карточку документа с черновиком в `ONLYOFFICE`
5. показать финальный документ в `NauDoc`

## 5. Что не стоит показывать как готовое

Чтобы не создать ложные ожидания, на показе не надо подавать как уже закрытое:

1. полноценный production и серверную зрелость
2. `SSO/LDAP`
3. прием заявок из `email` и мессенджеров
4. realtime-чат
5. полностью автоматическое создание реального документа в `NauDoc` без промежуточного интеграционного слоя
6. финальный боевой контур ЭЦП для всех сценариев

Это лучше формулировать как:

1. уже предусмотрено архитектурой
2. следующий этап развития
3. готово к внедрению поэтапно

## 6. Рекомендуемый сценарий показа

### Сценарий 1. Сотрудник

Показать:

1. вход в систему
2. личный рабочий экран
3. создание заявки на обслуживание
4. появление связанной карточки документа
5. статус документа в рабочем контуре

Ключевая мысль:

`пользователь работает в одном понятном веб-интерфейсе`

### Сценарий 2. Руководитель

Показать:

1. панель контроля
2. проекты и заявки на контроле
3. документные статусы
4. просрочки или риски

Ключевая мысль:

`руководитель видит не только задачи, но и документный статус процессов`

### Сценарий 3. Канцелярия / документооборот

Показать:

1. карточку документа
2. тип документа
3. статус маршрута
4. ссылку на документный контур
5. переход в `NauDoc`

Ключевая мысль:

`формальный документооборот не потерян и связан с рабочими процессами`

### Сценарий 4. Совместная работа

Показать:

1. поле `Рабочий черновик`
2. открытие документа в `ONLYOFFICE`
3. редактирование документа в браузере
4. связь черновика с карточкой документа

Ключевая мысль:

`совместная работа над документом вынесена в современный веб-редактор`

## 7. Как лучше упаковать платформу для заказчика

На показе систему лучше позиционировать так:

1. `Rukovoditel` — рабочая оболочка для сотрудников и руководителей
2. `NauDoc` — документное ядро и формальный маршрут документа
3. `middleware` — внутренняя интеграционная логика
4. `ONLYOFFICE` — совместная работа с документом в браузере

Для заказчика это должно звучать не как "четыре разных сервиса", а как:

`единая платформа с разделением ролей внутри архитектуры`

## 8. План быстрой подготовки к демонстрации

### Этап 1. Презентационный слой

1. почистить меню
2. переименовать сущности и статусы
3. подготовить красивые демо-данные
4. оформить стартовые dashboard-экраны

### Этап 2. Сценарный слой

1. собрать 3-4 готовых сценария
2. заранее связать записи между собой
3. подготовить 1-2 документа в `ONLYOFFICE`
4. подготовить 1-2 финальных документа в `NauDoc`

### Этап 3. Контроль показа

1. прогнать smoke-check перед демонстрацией
2. открыть вкладки заранее
3. не показывать технические экраны без необходимости
4. держать акцент на бизнес-процессе, а не на контейнерах и админке

### Этап 4. Повторяемая подготовка стенда

Перед каждым показом можно прогонять один сценарий подготовки:

```bash
cd /home/lebedeffson/Code/Документооборот/rukovoditel-test
bash prepare_customer_demo.sh
```

Он делает следующее:

1. обновляет брендирование и текст логин-экрана
2. приводит демо-данные к стабильному виду
3. догоняет синхронизацию заявок, проектов и карточек документов
4. сохраняет стенд в повторяемом состоянии перед показом

На текущий момент у нас уже есть не просто идея, а рабочий демонстрационный каркас платформы.

Его уже можно показывать заказчикам, если предварительно:

1. не пытаться показать все сразу

5. отдельно обозначить, что еще входит в следующий этап развития
