# План интеграции экосистемы ONLYOFFICE

## Цель

Интегрировать в текущую платформу три продукта ONLYOFFICE так, чтобы они дополняли существующий стек, а не дублировали его:

- `ONLYOFFICE Docs` — встроенный редактор документов и совместная работа
- `ONLYOFFICE DocSpace` — внешние комнаты и защищенное совместное взаимодействие
- `ONLYOFFICE Workspace` — ограниченный корпоративный сервисный контур

Центральный стек при этом сохраняется:

- `Rukovoditel` — рабочий кабинет, процессы, заявки, проекты, карточки документов
- `NauDoc` — официальный документный контур, регистрация, архив, маршруты, ЭЦП
- `middleware / Bridge` — интеграционный слой, синхронизация ссылок, статусов и метаданных

## Главный принцип

Интегрируем не “три отдельных портала”, а три специализированных слоя:

1. `Docs` — редактирование
2. `DocSpace` — внешнее сотрудничество
3. `Workspace` — только сервисные корпоративные функции

Если пустить `Workspace` в те же процессы, что уже ведет `Rukovoditel`, система начнет дублировать сама себя.

## Роли продуктов в нашей архитектуре

### 1. ONLYOFFICE Docs

Используем как встроенный редактор внутри основного пользовательского контура.

Назначение:

- редактирование документов в браузере
- совместная работа нескольких пользователей
- работа с черновиками
- редактирование из карточки документа

Где встраивается:

- `Карточки документов`
- затем `Проекты`
- затем `Заявки`
- затем при необходимости `База документов`

Что остается за `NauDoc`:

- регистрация
- официальный экземпляр документа
- согласование / утверждение / архив
- ЭЦП

### 2. ONLYOFFICE DocSpace

Используем как внешний контур совместной работы.

Назначение:

- внешние комнаты
- доступ подрядчиков / партнеров / комиссий
- безопасный обмен документами
- отдельные пространства под согласование или проверку
- временные совместные контуры вне внутреннего рабочего кабинета

Где используется:

- связанный контур для документа
- связанный контур для проекта
- связанный контур для внешнего согласования

### 3. ONLYOFFICE Workspace

Используем только как ограниченный сервисный слой.

Допустимые роли:

- почта
- календарь
- корпоративные коммуникации
- возможно intranet-панель или общий workplace

Что не переносим в него:

- заявки
- проекты
- карточки документов
- реестр документов
- основную процессную логику

То есть `Workspace` не должен становиться вторым `Rukovoditel`.

## Целевая пользовательская схема

### Для сотрудника

1. Заходит в основную платформу
2. Работает в `Rukovoditel`
3. Открывает документ в `ONLYOFFICE Docs`
4. При необходимости идет в `NauDoc` за официальным документом
5. Если есть внешняя совместная работа — переходит в связанную комнату `DocSpace`

### Для руководителя

1. Ведет работу через `Rukovoditel`
2. Видит статусы документов
3. Открывает черновик в `ONLYOFFICE Docs`
4. При необходимости открывает официальный документ в `NauDoc`
5. Для внешней комиссии или согласования использует `DocSpace`

### Для канцелярии

1. Получает готовый документ
2. Регистрирует и ведет официальный контур в `NauDoc`
3. Возвращает официальный статус и ссылку обратно в рабочий кабинет

## Единая архитектура интеграции

### Главная система

`Rukovoditel` остается основной стартовой точкой.

Из него пользователь должен видеть:

- рабочие модули
- документные карточки
- переход в `NauDoc`
- переход в `ONLYOFFICE Docs`
- переход в нужную комнату `DocSpace`

### Bridge / middleware

Интеграционный слой должен хранить связи между:

- `Rukovoditel entity`
- `Rukovoditel item`
- `ONLYOFFICE draft`
- `DocSpace room`
- `NauDoc document URL`
- статусами публикации / регистрации / подписания

### Источник истины

Чтобы не получить конфликт платформ:

1. процессы и контекст работы — `Rukovoditel`
2. официальный документ — `NauDoc`
3. рабочий черновик — `ONLYOFFICE Docs`
4. внешняя совместная работа — `DocSpace`
5. корпоративные сервисы — `Workspace`

## Этапы внедрения

## Этап 1. Закрепить ONLYOFFICE Docs как основной редакторный слой

Что делаем:

1. Доводим текущую интеграцию `ONLYOFFICE Docs` до production-логики
2. Закрепляем его в `Карточки документов`
3. Показываем draft/final distinction
4. Привязываем публикацию в `NauDoc`

Результат:

- документ готовится в браузере
- несколько пользователей редактируют его совместно
- официальный документ публикуется в `NauDoc`

## Этап 2. Добавить DocSpace как внешний collaboration-контур

Что делаем:

1. Поднимаем `DocSpace` рядом с платформой
2. Делаем единый вход или хотя бы единый переход
3. Добавляем в карточку документа поле/ссылку:
   - `Связанная комната DocSpace`
4. Добавляем сценарии:
   - внешнее согласование
   - обмен документами с подрядчиком
   - room под проект

Результат:

- внутренняя работа остается в платформе
- внешняя и смешанная работа уходит в rooms

## Этап 3. Подключить Workspace ограниченно

Что делаем:

1. Поднимаем `Workspace`
2. Используем только допустимые модули:
   - mail
   - calendar
   - portal/workplace
3. Не дублируем процессы `Rukovoditel`
4. На главной платформы добавляем только сервисный вход

Результат:

- работодатель получает “всю экосистему”
- но основная логика не распадается на несколько одинаковых систем

## Этап 4. Свести пользователей и роли

Что нужно:

1. единый каталог пользователей
2. единое сопоставление ролей
3. единый или полуединый вход
4. единые профильные ссылки между контурами

Минимум:

- синхронизация пользователей
- сопоставление ролей

Желательно:

- `SSO / LDAP / Keycloak`

## Этап 5. Свести статусы и документы

Bridge должен понимать и возвращать:

- статус черновика
- статус внешней комнаты
- статус публикации
- статус регистрации
- статус согласования
- статус подписи

Пример статусов:

- `draft`
- `coauthoring`
- `external_review`
- `ready_for_registration`
- `registered`
- `approved`
- `signed`
- `archived`

## Что потребуется сделать в коде

### В `Rukovoditel`

1. расширить карточки документов под `DocSpace`
2. вывести статусы экосистемы на главную и в карточки
3. добавить действия:
   - открыть в `ONLYOFFICE Docs`
   - открыть комнату `DocSpace`
   - открыть официальный документ `NauDoc`
4. добавить правила прав по ролям

### В `middleware`

1. расширить bridge-модель:
   - `onlyoffice_file_id`
   - `docspace_room_id`
   - `docspace_room_url`
   - `workspace_context`
2. добавить sync-правила для статусов
3. добавить возврат статусов в `Rukovoditel`

### В `NauDoc`

1. добавить мягкие действия перехода:
   - открыть черновик
   - открыть связанную внешнюю комнату
2. не делать глубокую встройку нового UI в старый Zope на первом этапе

### В `gateway`

1. опубликовать стабильные URL:
   - `/`
   - `/docs/`
   - `/office/`
   - `/docspace/`
   - `/workspace/`
   - `/bridge/`
2. обеспечить единый внешний домен

## Что не делаем

1. не переносим ядро процессов в `Workspace`
2. не делаем `DocSpace` основным внутренним рабочим кабинетом
3. не ломаем роль `NauDoc` как официального документного контура
4. не пытаемся сразу склеить все нативно в один монолит

## Практический порядок запуска

### Очередь 1

- довести `ONLYOFFICE Docs`
- привязать его к публикации в `NauDoc`
- вывести статусы в UI

### Очередь 2

- поднять `DocSpace`
- ввести room-based сценарии
- связать комнату с карточкой документа и проектом

### Очередь 3

- поднять `Workspace`
- подключить только сервисные модули
- не трогать процессное ядро

### Очередь 4

- единый каталог пользователей
- SSO
- роли и единая навигация

## Итоговая рекомендация

Для нашей платформы правильная интеграция выглядит так:

1. `ONLYOFFICE Docs` — обязательный встроенный редактор
2. `DocSpace` — внешний collaboration-контур
3. `Workspace` — ограниченный сервисный корпоративный слой

Это позволяет:

- выполнить требование “встроить все”
- не разрушить текущую архитектуру
- не получить дублирование процессов
- сохранить понятную роль каждой системы

## Следующий практический шаг

Следующим этапом берем в работу не все три продукта одновременно, а первый реальный слой:

1. закрепить `ONLYOFFICE Docs` как production-путь работы с документом
2. затем спроектировать модель `DocSpace room -> карточка документа / проект`
3. после этого подключать `Workspace` как сервисный портал
