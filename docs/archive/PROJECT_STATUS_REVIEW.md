# Обзор текущего состояния проекта

Дата фиксации состояния: `2026-03-24`

## 1. Короткий вывод

Проект уже перестал быть просто набором идей и тестовых контейнеров.

Сейчас у нас есть **рабочий гибридный стенд**:

1. `NauDoc` как legacy-контур документов
2. `Rukovoditel` как основной пользовательский рабочий слой
3. `middleware` как слой интеграции между ними

Это хорошая база для дальнейшего развития.

На текущем этапе проект лучше **не переписывать полностью**, а доводить до полноценной пилотной системы через:

1. улучшение интерфейса
2. расширение рабочих сущностей
3. автоматизацию интеграции
4. подготовку серверного и пользовательского контура к пилоту

## 2. Что есть сейчас

### 2.1 Живой стенд

На момент проверки подняты и работают:

1. единый внешний вход: `https://localhost:18443/`
2. `NauDoc`: `https://localhost:18443/docs/`
3. `Rukovoditel`: `https://localhost:18443/`
4. `Middleware`: `https://localhost:18443/bridge/`

Healthcheck `middleware` возвращает `ok`, обе системы доступны.

### 2.2 Архитектура

Текущая схема уже соответствует нашей целевой модели:

`Rukovoditel <-> Middleware <-> NauDoc`

Роли контуров:

1. `NauDoc`
   - документы
   - версии
   - согласование
   - подписание
   - архив
2. `Rukovoditel`
   - проекты
   - задачи
   - заявки
   - карточки документов
   - МТЗ
   - рабочие представления для сотрудников и руководителей
3. `Middleware`
   - хранение связей
   - статусы интеграции
   - возврат ссылок и статусов обратно в рабочий контур

### 3.3 По интеграции

Интеграционный слой уже не пустой.

Что реализовано:

1. `middleware` хранит связи между сущностями `Rukovoditel` и документным контуром
2. поддерживается промежуточный статус `pending_nau_doc`
3. есть `JSON API`
4. есть web-UI для ручной проверки и правки связей
5. есть сценарий:
   - заявка
   - автосоздание карточки документа
   - публикация bridge-записей
   - возврат финальной ссылки и статуса обратно в `Rukovoditel`
6. есть сценарий:
   - проект
   - автосоздание карточки документа
   - публикация bridge-записей
   - возврат финальной ссылки обратно в карточку проекта
7. есть фоновый worker, который периодически запускает sync-циклы
8. у проектов и заявок есть видимое поле `Статус документа / интеграции`
9. в `middleware` есть быстрые фильтры и журнал незавершенных связей
10. в `Rukovoditel` есть отчеты по сущностям без финального документа

### 3.4 Что проверено руками

На живом стенде уже проверен round-trip сценарий:

1. создана тестовая заявка
2. создана связанная карточка документа
3. в `middleware` сохранена связь
4. в bridge проставлен финальный `NauDoc URL`
5. обратная синхронизация записала изменения в `Rukovoditel`

Фактически в тесте уже отработали:

1. заявка получила статус `Выполнена`
2. карточка документа получила статус `Подписан`
3. обе записи получили ссылку на `NauDoc`

## 4. Что сейчас работает хорошо

### 4.1 Сильные стороны текущего решения

1. Есть реальный рабочий каркас, а не только документы и планы.
2. Не потерян legacy-функционал документооборота.
3. Пользовательская часть вынесена в более удобный контур.
4. Интеграция уже начата технически правильно, через отдельный слой.
5. Дальнейшее развитие можно делать поэтапно, без полной переписки всего проекта.

### 4.2 Почему это удачная база

1. Мы не ломаем `NauDoc`, где уже живет документная логика.
2. Мы не заставляем пользователя жить внутри старого интерфейса.
3. Мы можем быстро добавлять новые сущности и процессы.
4. Мы можем постепенно автоматизировать интеграцию.

## 6. Главные ограничения и риски

### 6.1 Legacy-риск

`NauDoc` остается на старом стеке, и это нужно держать в голове:

1. он полезен как ядро документооборота
2. но неудобен как главный пользовательский интерфейс
3. и рискован как место для активной большой доработки

### 6.2 Интеграционный риск

Пока часть сценариев требует ручного или полуавтоматического шага.

Значит, до пилота обязательно нужно закрыть:

1. регламенты статусов
2. правила синхронизации
3. обработку ошибок

### 6.3 UX-риск

Если не довести роли, меню и карточки, пользователи будут воспринимать систему как две разных программы.

Это критично, потому что нам нужен эффект:

1. одной точки входа
2. одной понятной логики работы

## 7. Что делать по стеку

Текущая рекомендация остается прежней:

1. **не переписывать все с нуля сейчас**
2. **довести гибридную схему до пилота**
3. **о полном переписывании думать только после пилота**

Почему:

1. уже есть рабочее ядро документооборота
2. уже есть более удобная пользовательская оболочка
3. уже есть интеграционный слой
4. сейчас быстрее и выгоднее довести систему до реальной эксплуатации, чем уходить в длинную миграцию

## 10. Definition of Ready для пилота

Считаем систему готовой к пилоту, когда:

1. пользователь может подать заявку и увидеть ее статус
2. ответственный может работать с задачей и карточкой документа
3. руководитель видит контрольный экран
4. документ связан с карточкой и доступен по ссылке
5. переходы статусов понятны и воспроизводимы
6. есть базовые инструкции
7. серверный контур можно перезапустить и восстановить

## 11. Рекомендуемое направление

Самый здравый путь сейчас такой:

1. продолжать развивать **гибридную систему**
2. не трогать большой перепиской весь legacy-контур
3. улучшать `Rukovoditel` как основную рабочую среду
4. усиливать `middleware` как настоящий интеграционный центр
5. довести систему до пилота и только потом принимать решение о большой миграции
