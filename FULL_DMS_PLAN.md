# План развития полноценной системы документооборота

## 1. Цель

Собрать из текущего стенда `NauDoc + Rukovoditel + middleware` полноценную рабочую систему, в которой:

1. сотрудники удобно работают каждый день
2. руководители видят контроль, сроки и статус исполнения
3. документы живут по формальному маршруту: карточка -> согласование -> подписание -> ознакомление -> архив
4. серверная часть работает согласованно и предсказуемо
5. пользователям не нужно ставить тяжелый локальный клиент

## 2. Базовое решение

На текущем этапе **не переписываем все с нуля**.

Используем такую модель:

1. `NauDoc` остается документным ядром:
   - документы и версии
   - согласование и утверждение
   - регистрация
   - архив
   - legacy-маршруты
2. `Rukovoditel` становится основной рабочей оболочкой:
   - проекты
   - поручения
   - заявки
   - карточки
   - обсуждения
   - управленческие экраны
3. `middleware` становится единым слоем интеграции:
   - связка записей
   - синхронизация статусов
   - единые ссылки на документы
   - API для дальнейших интеграций

## 3. Что нужно получить на выходе

### 3.1 Пользовательский контур

Пользователь должен работать через **единый веб-интерфейс**, без сложной установки на ПК.

Целевой принцип:

1. вход по одной ссылке
2. понятное меню
3. формы под реальные процессы
4. быстрый поиск
5. прямой переход к связанному документу
6. скачивание файла и просмотр вложений без лишних шагов

Практически это значит:

1. основная ежедневная работа идет через `Rukovoditel`
2. документы открываются по связанной ссылке в `NauDoc`
3. если нужна ЭЦП, локальный компонент ставится только там, где это действительно необходимо

### 3.2 Руководительский контур

Руководитель должен видеть:

1. сколько документов на согласовании
2. сколько задач просрочено
3. какие заявки не исполнены
4. какие проекты требуют внимания
5. какие документы готовы к подписанию или архивированию

### 3.3 Серверный контур

Серверная часть должна работать как единая система:

1. сервисы запускаются контейнерами
2. у каждого сервиса есть понятная зона ответственности
3. есть бэкапы
4. есть журнал ошибок и мониторинг
5. есть единый reverse proxy
6. есть тестовый и продовый контур

## 4. Целевая функциональная модель

### 4.1 Электронная канцелярия

Нужно обеспечить полный цикл:

1. создание карточки документа
2. создание файла документа
3. маршрутизация по типу документа
4. согласование
5. утверждение
6. подписание
7. ознакомление
8. регистрация
9. завершение действия
10. архивирование

Где это живет:

1. карточка и рабочая оболочка могут жить в `Rukovoditel`
2. документ, версия, архив и часть маршрутов живут в `NauDoc`
3. связь и синхронизация живут в `middleware`

### 4.2 Работа с проектами

Нужно:

1. карточка проекта
2. этапы
3. задачи и поручения
4. сроки
5. ответственные
6. связанные документы
7. контроль исполнения

Где это живет:

1. основной модуль в `Rukovoditel`
2. документы проекта через ссылки на `NauDoc`

### 4.3 Работа с заявками на обслуживание

Нужно:

1. прием заявки
2. регистрация
3. классификация
4. назначение исполнителя
5. SLA и сроки
6. контроль исполнения
7. закрытие
8. история переписки и вложений

Источники заявок:

1. веб-форма
2. email
3. позже: мессенджер или бот

Где это живет:

1. основной модуль в `Rukovoditel`
2. интеграционные входы через `middleware`

### 4.4 База документов

Нужно:

1. единый каталог документов
2. категории и типы
3. реквизиты
4. версии
5. теги
6. фильтры и поиск
7. доступ по ролям

### 4.5 Обсуждения / чат

На первом этапе не нужен тяжелый realtime-чат.

Практичное решение:

1. обсуждения в карточках и задачах в `Rukovoditel`
2. уведомления по событиям
3. потом при необходимости:
   - отдельный чат-модуль
   - или интеграция с корпоративным мессенджером

### 4.6 ЭЦП

Нужно разделить две зоны:

1. маршрут документа и его состояние
2. фактическое подписание через ЭЦП

Решение:

1. логику документа оставляем в `NauDoc`
2. ЭЦП внедряем по отдельному потоку
3. локальные компоненты ставим только тем пользователям, кто реально подписывает

### 4.7 МТЗ

Нужно:

1. заявки на МТЗ
2. согласование потребности
3. статус обеспечения
4. связь с проектом или подразделением
5. базовая отчетность

На старте делаем именно **учет заявок и движения**, а не полный складской ERP.

## 5. Какой пользовательский опыт делаем

### 5.1 Принцип

Для сотрудников система должна быть `zero-install`:

1. открывается в браузере
2. работает по одной ссылке
3. не требует отдельной тяжелой установки
4. при желании можно выдать ярлык на рабочий стол

### 5.2 Что обязательно делаем для удобства

1. единое меню по ролям
2. упрощенные формы без лишних полей
3. понятные статусы на русском
4. цветовые маркеры сроков и просрочки
5. быстрые кнопки:
   - создать заявку
   - создать поручение
   - создать карточку документа
6. личный кабинет сотрудника
7. отдельный кабинет руководителя

### 5.3 Что не делаем на старте

1. толстый desktop-клиент
2. сложный локальный агент для всех пользователей
3. двойной ввод в двух системах вручную

## 6. Серверная архитектура

### 6.1 Контейнеры

Текущий логичный состав:

1. `reverse proxy`
2. `Rukovoditel`
3. `MariaDB`
4. `middleware`
5. `NauDoc legacy`

### 6.2 Что должно быть дополнительно

1. общий внешний URL
2. внутренние сервисные URL
3. резервное копирование:
   - БД `MariaDB`
   - `bridge.db`
   - данные `NauDoc`
4. логи
5. мониторинг доступности
6. обновляемые конфиги

### 6.3 Что делаем по доступу

1. внешне пользователю показываем один портал
2. доступ к `NauDoc` можно оставлять через прокси и прямые ссылки из `Rukovoditel`
3. админские сервисы не открываем наружу напрямую

### 6.4 Минимальный прод-контур

1. `Nginx` или `Traefik`
2. `HTTPS`
3. Docker Compose
4. отдельные volume под данные
5. ежедневный backup
6. staging-контур перед продом

## 7. Роли и зоны ответственности в системе

### 7.1 Сотрудник

1. подает заявки
2. создает карточки
3. прикладывает файлы
4. смотрит свои задачи и документы

### 7.2 Исполнитель

1. берет заявку или поручение
2. меняет статус
3. пишет результат
4. прикладывает итоговые файлы

### 7.3 Руководитель / преподаватель

1. согласует
2. утверждает
3. контролирует сроки
4. смотрит просрочку и загрузку

### 7.4 Канцелярия / делопроизводитель

1. регистрирует документы
2. ведет карточки
3. контролирует маршруты
4. переводит в архив

### 7.5 Администратор

1. роли
2. справочники
3. маршруты
4. интеграции
5. резервирование и эксплуатация

## 8. Что нужно реализовать по этапам

4. привести меню к рабочему виду
5. описать словарь статусов

Результат:

1. система уже выглядит как единый продукт
2. пользователю понятно, куда идти

1. проекты

1. повседневная работа уже может идти в `Rukovoditel`

## 9. Что важно не упустить

### 9.1 Главное правило

Нельзя делать так, чтобы пользователи:

1. создавали одну и ту же сущность дважды
2. не понимали, где “главная правда”
3. терялись между двумя интерфейсами

Поэтому:

1. ежедневная работа должна быть через `Rukovoditel`
2. документный контур должен быть прозрачно связан с `NauDoc`
3. интеграция должна скрывать техническую сложность

### 9.2 Второе главное правило

Не строим сразу “идеальную универсальную платформу”.

Сначала делаем:

1. понятные типовые процессы
2. работающие маршруты
3. удобные экраны
4. реальную эксплуатацию

И только потом:

1. сложные интеграции
2. deep customization
3. большой рефакторинг стека

## 10. Итоговое рекомендуемое решение

### На сейчас

1. не переписываем весь `NauDoc`
2. развиваем `Rukovoditel` как основной рабочий портал
3. превращаем `middleware` в реальный интеграционный слой
4. делаем единый пользовательский вход
5. минимизируем локальную установку для пользователей

### Целевой результат

Получаем систему, где:

1. сотрудник работает через браузер
2. руководитель видит контроль и статусы
3. документы проходят формальный маршрут
4. серверная часть развертывается предсказуемо
5. обе системы работают согласованно, а не как два разрозненных продукта
