# Сценарий показа платформы заказчику

Дата актуальности: `2026-03-26`

## 1. Цель показа

Показать, что платформа уже закрывает основной рабочий контур:

1. сотрудники работают в понятном веб-интерфейсе
2. руководитель видит статус процессов и документов
3. документы связаны с проектами и заявками
4. совместная работа над документом идет в браузере через `ONLYOFFICE Docs`
5. официальный документный контур сохраняется в `NauDoc`

## 2. Что подготовить до встречи

Открыть заранее:

1. `https://localhost:18443/`
2. `https://localhost:18443/docs/`

Под рукой держать логины:

1. `admin / пароль из DOCFLOW_ADMIN_PASSWORD`
2. `manager.test / пароль из DOCFLOW_ROLE_DEFAULT_PASSWORD`
3. `office.test / пароль из DOCFLOW_ROLE_DEFAULT_PASSWORD`
4. для `NauDoc`: логин `admin`, пароль брать из `NAUDOC_PASSWORD` в корневом `.env`

Если нужно быстро привести стенд в порядок:

```bash
cd /home/lebedeffson/Code/Документооборот/rukovoditel-test
bash prepare_hospital_baseline.sh
```

## 3. Структура показа

Рекомендуемая длительность: `7-12 минут`

Порядок:

1. единый вход
2. рабочий контур сотрудника
3. контроль руководителя
4. карточка документа
5. совместное редактирование в `ONLYOFFICE`
6. официальный документ в `NauDoc`
7. дополнительный процесс: МТЗ

## 4. Сценарий показа по шагам

### Шаг 1. Показать главный вход

Открыть:

1. `https://localhost:18443/`

Сказать:

1. это единый вход в рабочий контур
2. сотрудники и руководители работают здесь ежедневно
3. документный контур и совместное редактирование встроены в общую платформу

Акцент:

1. пользователю не нужно начинать работу в legacy-интерфейсе

### Шаг 2. Показать рабочий контур

Войти под:

1. `admin / пароль из DOCFLOW_ADMIN_PASSWORD`

Открыть разделы:

1. `Работа`
2. `Документы`
3. `Контроль`

Сказать:

1. здесь собраны проекты, заявки, карточки документов, обсуждения и МТЗ
2. роли уже разведены по меню и рабочим экранам

Акцент:

1. один интерфейс для ежедневной работы
2. понятная навигация для сотрудника и руководителя

### Шаг 3. Показать заявку, связанную с документом

Открыть:

1. `Заявки на обслуживание`
2. запись `Подготовка маршрута согласования приказа о пилоте`

Показать поля:

1. `Связанная карточка документа`
2. `Ссылка на NauDoc`
3. `Статус документа / интеграции`

Сказать:

1. заявка не живет отдельно от документа
2. по рабочему кейсу видно, дошел ли документ до финального состояния

Акцент:

1. связка процесса и документа
2. руководитель видит не просто заявку, а статус официального документа

### Шаг 4. Показать проектный контур

Открыть:

1. `Проекты и инициативы`
2. запись `Внедрение единой платформы документооборота`

Показать:

1. `Связанная карточка документа`
2. `Ссылка на NauDoc`
3. `Статус документа / интеграции`

Потом открыть второй проект:

1. `Подготовка регламента согласования учебных документов`

На нем показать:

1. проект в работе
2. документ еще ожидается

Акцент:

1. система показывает не только завершенные кейсы, но и незакрытые
2. можно управлять рисками по документам внутри проектного контура

### Шаг 5. Показать карточку документа

Открыть:

1. `Карточки документов`
2. запись `Служебная записка о запуске пилотного контура`

Показать:

1. тип документа
2. статус документа
3. ссылку на `NauDoc`
4. блок `Быстрые действия с документом`

Сказать:

1. карточка документа связывает рабочий контур и официальный контур
2. из нее видно, где документ редактируется и где хранится его официальная версия

### Шаг 6. Показать совместную работу в ONLYOFFICE

1. открыть карточку документа
2. нажать `Открыть документ в редакторе`

Сказать:

1. черновик редактируется прямо в браузере
2. не нужен отдельный desktop-редактор
3. это основа для совместной работы нескольких пользователей над одним документом

Акцент:

1. современный web-редактор внутри платформы
2. удобство для удаленной работы и мобильного сценария

### Шаг 7. Показать официальный контур в NauDoc

Из карточки документа нажать ссылку на `NauDoc` или открыть:

1. `https://localhost:18443/docs/`

Если попросит логин:

1. логин `admin`
2. пароль брать из `NAUDOC_PASSWORD` в корневом `.env`

Показать:

1. официальный документный контур
2. хранилище
3. системные справочники
4. что документный контур работает отдельно и стабильно

Акцент:

1. формальный документооборот не потерян
2. архив, регистрация и маршрут документа сохраняются

### Шаг 8. Показать дополнительный процесс МТЗ

Открыть:

1. `Заявки на МТЗ`
2. запись `Оснащение канцелярии МФУ и расходными материалами`

Сказать:

1. платформа уже покрывает не только документы, но и обеспечивающие процессы
2. это важно для единой операционной среды организации

## 5. Что говорить как преимущества системы

### 5.1 Для сотрудников

1. понятный веб-интерфейс
2. проекты, заявки и документы связаны между собой
3. документы можно редактировать в браузере
4. не нужно искать информацию в разных системах вручную

### 5.2 Для руководителей

1. видно статус процесса и статус документа одновременно
2. легко контролировать проекты и заявки
3. есть отдельный контур контроля и отчетов
4. снижается разрыв между исполнительской работой и формальным документооборотом

### 5.3 Для канцелярии

1. не ломается официальный документный контур
2. сохраняются карточки, регистрация, маршруты и архив
3. рабочие процессы не живут отдельно от документа

### 5.4 Для организации в целом

1. гибридная архитектура уже работает
2. legacy-документооборот сохранен
3. современный UX вынесен в рабочий слой
4. совместное редактирование добавлено без переписывания всей системы
5. платформа расширяется поэтапно

## 6. Что не стоит заявлять как уже полностью закрытое

Не подавать как полностью завершенное:

1. production-инфраструктуру
2. `SSO/LDAP`
3. прием заявок из почты и мессенджеров
4. финальный промышленный контур `ЭЦП` для всех сценариев
5. realtime-чат

Правильная формулировка:

1. архитектурно предусмотрено
2. следующий этап развития
3. можно внедрять поэтапно
