# План внедрения ONLYOFFICE в платформу

## Цель

Добавить в текущую платформу браузерное редактирование документов и совместную работу нескольких пользователей без переписывания `NauDoc`.

Целевой контур:

- `Rukovoditel` — рабочее место пользователя и редакторный слой
- `ONLYOFFICE Docs` — веб-редактор и co-authoring
- `middleware` — синхронизация связей, статусов и публикации
- `NauDoc` — официальный контур документа, регистрации, согласования, архива и ЭЦП

## Рекомендуемый первый пилот

Первой точкой внедрения берем сущность:

- `Карточки документов`

Почему именно она:

1. это уже центральная документная сущность рабочего контура
2. она уже связана с `NauDoc`
3. через нее потом проще распространить редактор на:
   - `Заявки на обслуживание`
   - `Проекты и инициативы`

## План внедрения 1–10

### 1. Поднять `ONLYOFFICE Docs` как отдельный сервис рядом со стеком

Что делаем:

- добавляем отдельный контейнер `onlyoffice/documentserver`
- подключаем его в общий gateway-контур
- фиксируем публичный URL редактора

Результат:

- редактор доступен из браузера
- `api.js` доступен по стабильному URL

## 2. Настроить сетевую схему и callback-маршрут

Что делаем:

- проверяем, что `Rukovoditel` может отдавать файл `ONLYOFFICE`
- проверяем, что `ONLYOFFICE` может достучаться до callback URL в `Rukovoditel`
- выравниваем public/base URL за gateway

Результат:

- открытие документа работает
- сохранение обратно по callback работает

## 3. Исправить локальные проблемы текущего коннектора

Что делаем:

- убрать жесткий `type = desktop`
- выбрать `desktop/mobile/embedded` динамически
- исправить ошибку `lang/region`
- проверить JWT-конфиг

Файлы:

- [onlyoffice_editor.php](/home/lebedeffson/Code/Документооборот/rukovoditel-test/dist/modules/items/actions/onlyoffice_editor.php)
- [fieldtype_onlyoffice.php](/home/lebedeffson/Code/Документооборот/rukovoditel-test/dist/includes/classes/fieldstypes/fieldtype_onlyoffice.php)

Результат:

- редактор корректно открывается в вебе
- мобильный и десктопный режимы не ломаются

## 4. Добавить pilot-field в `Карточки документов`

Что делаем:

- создаем поле типа `fieldtype_onlyoffice`
- добавляем его в сущность `Карточки документов`
- задаем допустимые форматы
- задаем правило редактирования по ролям

Результат:

- в карточке документа появляется рабочий редакторный файл

## 5. Настроить права редактирования

Что делаем:

- редактирование только для нужных ролей
- view/edit через `users_edit_access` или `assigned_users`
- отдельная логика для сотрудника, руководителя и канцелярии

Результат:

- документ не может редактировать кто угодно
- совместная работа идет в рамках ролей и назначений

## 6. Прогнать первый end-to-end сценарий

Сценарий:

1. создать карточку документа
2. загрузить `docx`
3. открыть в двух браузерах под разными пользователями
4. изменить документ одновременно
5. закрыть редактор
6. убедиться, что callback сохранил финальный файл

Результат:

- подтвержден реальный co-authoring
- подтверждено сохранение в системе

## 7. Привязать draft-контур к `middleware`

Что делаем:

- добавить в `middleware` метаданные по draft-файлу
- связать:
  - `document_card_id`
  - `onlyoffice_file_id`
  - статус черновика
  - ссылка на `NauDoc`
- подготовить статусный мост:
  - `draft`
  - `ready_for_registration`
  - `registered`
  - `approved`
  - `signed`

Результат:

- редакторный файл становится частью общей интеграции, а не отдельной жизнью

## 8. Сделать публикацию из черновика в официальный документный контур

Что делаем:

- определяем, какой файл считается официальной версией
- добавляем действие `Передать в NauDoc`
- сохраняем связь между draft и официальным документом
- возвращаем ссылку на финальный документ обратно в `Rukovoditel`

Результат:

- черновик и официальный документ не смешиваются
- путь пользователя становится понятным

## 9. Доработать UX

Что делаем:

- кнопка `Открыть в ONLYOFFICE`
- индикация:
  - черновик
  - в совместной работе
  - готов к регистрации
  - зарегистрирован в `NauDoc`
  - подписан
- менеджерский вид: кто редактировал и когда
- мягкая интеграция из `NauDoc`: кнопка перехода к связанному черновику

Результат:

- система ощущается единой
- пользователю не нужно понимать внутреннюю архитектуру

## 10. Расширить на заявки и проекты

После успешного пилота на `Карточки документов` переносим механику на:

- `Заявки на обслуживание`
- `Проекты и инициативы`

Результат:

- документы можно коллективно готовить прямо в рабочих процессах
- дальше уже можно расширять на шаблоны, регламенты и базу документов

## Что не делаем на первом этапе

1. не переписываем `NauDoc`
2. не строим глубокую нативную встройку редактора в старый Zope UI
3. не тащим `ONLYOFFICE Workspace` как второй основной портал
4. не начинаем с `DocSpace`, пока не закрыт внутренний документный сценарий

## Критерии успешного пилота

Пилот считается успешным, если:

1. документ открывается в браузере
2. два пользователя одновременно редактируют один файл
3. изменения сохраняются обратно
4. карточка документа хранит актуальную редакторную версию
5. пользователь видит ссылку на официальный документ в `NauDoc`
6. роли не позволяют редактировать документ всем подряд

### Волна 2

- расширить на заявки и проекты

## Итоговая рекомендация

Начинаем не с большого продукта `Workspace`, а с точечной и сильной интеграции `ONLYOFFICE Docs` в наш текущий гибридный контур.

Это даст:

- совместное редактирование в вебе
- Linux-friendly разворачивание
- минимальную ломку текущей архитектуры
- понятный путь к мобильному сценарию
- сохранение `NauDoc` как официального СЭД-контура
