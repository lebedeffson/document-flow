# План модернизации стека и интерфейса

## Короткий вывод

На текущем этапе **не стоит переписывать всю систему целиком**.

Более рациональный путь:

1. Оставить `NauDoc` как контур документооборота, архива, версий и согласования.
2. Использовать `Rukovoditel` как основной рабочий интерфейс для сотрудников и руководителей.
3. Развивать `middleware` как слой интеграции между системами.
4. Улучшать внешний вид, роли, сценарии работы и автоматизацию уже сейчас.
5. К вопросу полного переписывания возвращаться только после стабилизации процессов и пилота.

## Почему не переписывать все сразу

### `NauDoc`

- стек очень старый: `Python 2.x + Zope 2.x`
- логика документооборота уже есть и работает
- полное переписывание с сохранением всех нюансов займет долго
- высокий риск потерять важные детали маршрутов, согласования и архивной логики

### `Rukovoditel`

- стек заметно живее и проще в сопровождении
- удобно моделирует сущности, процессы, заявки, проекты и карточки
- подходит как слой повседневной работы для пользователей
- может быть постепенно “облагорожен” по интерфейсу и логике без остановки системы

### `Middleware`

- это правильная точка для интеграции
- через него можно делать:
  - ссылки на документы
  - синхронизацию статусов
  - создание карточек
  - обмен метаданными
  - дальнейшие API-интеграции

## Рекомендуемая целевая схема

### Что оставляем

- `NauDoc`:
  - документы и версии
  - согласование и утверждение
  - регистрация
  - архив
  - юридически значимый контур

### Что переносим в основной пользовательский слой

- `Rukovoditel`:
  - проекты и инициативы
  - поручения и задачи
  - заявки на обслуживание
  - карточки документов
  - обсуждения
  - заявки на МТЗ
  - управленческие представления

### Что связывает оба контура

- `middleware`:
  - ссылки `Rukovoditel <-> NauDoc`
  - синхронизация статусов
  - журнал интеграционных операций
  - единые точки обмена данными

## Решение по стеку

### Вариант 1. Не переписывать сейчас

Это рекомендуемый вариант на ближайший этап.

Что делаем:

1. Оставляем текущий стек как есть.
2. Улучшаем интерфейс.
3. Донастраиваем сущности и роли.
4. Добавляем автоматизацию через `middleware`.
5. Доводим до пилота.

Оценка:

- самый быстрый путь к рабочей системе
- минимальный риск
- максимальная отдача в короткий срок

### Вариант 2. Частичная модернизация без полного переписывания

Это следующий логичный этап после пилота.

Что делаем:

1. Оставляем `NauDoc` только как legacy-контур документов.
2. Все новые функции делаем вне него.
3. `middleware` превращаем в полноценный интеграционный сервис.
4. Поверх при необходимости делаем современный frontend для руководителей.

Оценка:

- хороший баланс между скоростью и качеством
- не ломает существующую бизнес-логику
- позволяет постепенно “вытаскивать” функции из legacy

### Вариант 3. Полное переписывание

Пока не рекомендую.

Когда к нему стоит возвращаться:

1. процессы уже устоялись
2. понятен реальный целевой функционал
3. есть подтвержденный бюджет и команда
4. завершен пилот гибридной схемы

Оценка по сроку:

- `9-14 месяцев` и более

Основной риск:

- можно потратить много времени на перенос того, что уже работает, вместо запуска полезной системы для пользователей

## Что уже лучше делать сейчас

### Интерфейс

1. Освежить тему `Rukovoditel`.
2. Сделать рабочие разделы более чистыми и управленческими.
3. Настроить удобные формы для сотрудников.
4. Сделать отдельные представления для руководителей и преподавателей.

### Логика

1. Формализовать жизненный цикл заявок.
2. Связать проект, поручение и документ.
3. Привязать карточки к `NauDoc`.
4. Добавить управленческие фильтры и контроль сроков.

### Интеграция

1. Автоматическое создание карточки или ссылки в `middleware`.
2. Передача статусов между системами.
3. Подготовка API-точек под форму, почту или мессенджер.

## Практический план по этапам

### Этап 1. Быстрое улучшение без смены стека

Срок: `1-3 недели`

1. Обновить UI.
2. Довести меню и роли.
3. Собрать удобные карточки сущностей.
4. Подготовить базовые dashboard-представления.

### Этап 2. Рабочая интеграция процессов

Срок: `3-6 недель`

1. Синхронизация статусов.
2. Автоматическое связывание сущностей.
3. Обязательные поля и контроль маршрутов.
4. Отчеты для руководителей.

### Этап 3. Пилот

Срок: `4-8 недель`

1. Реальные пользователи.
2. Реальные сценарии заявок и документов.
3. Сбор замечаний.
4. Доработка интерфейса и логики.

### Этап 4. Решение о большой миграции

Срок: после пилота

По итогам принимаем решение:

1. оставляем гибридную архитектуру как основную
2. или начинаем постепенный вынос функций из `NauDoc`

## Рекомендуемое решение на сейчас

На сейчас предлагаю считать базовым такой подход:

1. **Стек не переписываем целиком.**
2. **Интерфейс и рабочие сценарии обновляем сразу.**
3. **`Rukovoditel` делаем основной системой для повседневной работы.**
4. **`NauDoc` оставляем как документный и архивный контур.**
5. **Интеграцию и новые функции развиваем через `middleware`.**

Это самый реалистичный путь, чтобы:

- быстро получить удобную систему
- не потерять существующий функционал
- не уйти в слишком длинную и дорогую переписку legacy
